## Why a patch (not minor)

These are four targeted bugfixes / UX polish items. No new tabs, no new system services, no new IPC surface beyond what already exists. Strict patch territory.

The four items:
- **Fix A** — Description edit in timesheet doesn't reflect locally on next view (saves to ClickUp but local list shows the old text).

## Fix A — Timesheet description edit doesn't reflect locally

**Today's behavior**: I edit a time entry's description, blur the textarea → `useTimeEntries.save` calls `window.helm.updateTimeEntry`. The server-side update lands in ClickUp (verified by refreshing ClickUp web). But when I click the entry again in Helm, the description field shows the **old** text. A full Refresh fixes it; a tab switch round-trip does not.

**Root cause** (`src/hooks/useTimeEntries.ts`, the `save` callback's defensive merge):

```typescript
const next: TimeEntry = {
  ...e,
  description: updated.description ?? e.description,  // ← bug
  ...
};
```

The merge falls back to **`e.description`** (the previous local value) when ClickUp's PUT response doesn't include the description field. ClickUp's response often omits description on a partial PUT — that's why the merge exists. But `e` is the *pre-patch* state, so the merge silently drops the user's edit.

The user's typed value lives in `patch.description`. The merge consults `updated` and `e` but never `patch`.

**Goal**: After saving, the local state reflects the user's just-typed value, regardless of what ClickUp's response includes.

### Sketch of the approach

In `useTimeEntries.save`, prefer the patch value over both the server response and the previous local value for any field the user explicitly set:

```typescript
// Pseudocode
const next: TimeEntry = {
  ...e,
  description: patch.description ?? updated.description ?? e.description,
  start:
    patch.start !== undefined
      ? patch.start
      : Number.isFinite(updated.start) && updated.start > 0
      ? updated.start
      : e.start,
  end: patch.end !== undefined ? patch.end : updated.end ?? e.end,
  duration: patch.duration ?? (Number.isFinite(updated.duration) && updated.duration > 0 ? updated.duration : e.duration),
  taskId: updated.taskId || e.taskId,
  taskName: updated.taskName || e.taskName,
};
```

The change is one-file, and the existing merge contract (defensive against missing server fields) is preserved — we just prioritize the user's intent first.

### Things to watch

- **Start/end timestamps**: the user's input is already a timestamp (from `fromLocalDatetimeInput`). Server may also normalize. Prefer the user's value.
- **Duration**: the renderer computes duration from `(end - start)` in `TimeEntryDetail.onBlur`. Trust the patch value if set.
- **No regression for `viaSpace` or other fields not in the patch**: those continue to fall back through `updated` → `e`.

### What likely changes

| Path | Change |
|---|---|
| `src/hooks/useTimeEntries.ts` | Reorder merge precedence to `patch` → `updated` → `e` for description / start / end / duration. |

## Idea 1 — "Open in ClickUp" on timesheet entry detail

**Today's behavior**: The Tasks tab's `TaskDetail` opens the task in ClickUp via `window.helm.openExternal(detail.url)` (`src/components/TaskDetail.tsx:249`). The timesheet's `TimeEntryDetail` has no equivalent — to jump to ClickUp from a timesheet entry, I have to switch to Tasks, find the task, then click open.

**Goal**: A small "Open in ClickUp" button in the panel-header of `TimeEntryDetail` that opens the entry's task page in ClickUp.

### Sketch of the approach

Time entries have a `taskId` but no `url` field on the `TimeEntry` type. The user's task list (`tasks` in App.tsx) does carry `url` per task. Two options:

**Option A — Look up via tasks list (simpler)**:
- Pass `tasks` (or just a `Map<taskId, url>`) into `TimeEntryDetail` as a prop.
- When the entry has a `taskId` matching a task in the list, render the button. Otherwise hide it.
- Reuses existing `tasks` already loaded in App.tsx; zero new IPC.

**Option B — Add `url` to TimeEntry**:
- Extend `TimeEntry` shape and `normalizeEntry` in `electron/services/clickup.ts` to capture the task URL from ClickUp's time-entry response.
- Always available even for entries whose task isn't in the user's `tasks` list (e.g. archived).

Lean **A** for 0.4.1 — minimal touch, covers ~99% of cases, no migration. If users hit "task not in list" cases, escalate to B.

### What likely changes

| Path | Change |
|---|---|
| `src/App.tsx` | Pass `tasks` (or a derived `Map<taskId, url>`) into `TimeEntryDetail`. |
| `src/components/TimeEntryDetail.tsx` | New optional prop `taskUrl: string \| null`. Render button next to "running" pill or in a corner of the panel-header. |

### Open questions

- Button placement: panel-header right side (matches Tasks pattern), or inline among the pills? Lean panel-header.
- Hide vs. disable when no URL? Hide — less visual noise.

## Idea 2 — Click the running task in the header to jump to it in Tasks

**Today's behavior**: `TimerBar` shows the running task name centered in the header. It's a static `<div>` — clicking it does nothing.

**Goal**: Click → switch to the Tasks tab → select the running task in TaskList → middle column shows TaskDetail for it. Same affordance as clicking a task name from anywhere else in the app.

### Sketch of the approach

`TimerBar` already receives `state` (TimerState). Add an `onTaskClick: (taskId: string) => void` prop. App.tsx wires it:

```typescript
// Pseudocode in App.tsx
<TimerBar
  state={timer}
  elapsedMs={elapsedMs}
  onStop={() => stop()}
  onTaskClick={(id) => {
    setTab('tasks');
    setSelected(id);
  }}
  lexicon={lexicon}
/>
```

In TimerBar.tsx, when `state.running && state.taskId`, wrap the task-name `<div>` in a button (or make it `cursor-pointer` + onClick). Idle state: no click handler.

Visual: subtle hover affordance (underline or bg-panelHi) so it's discoverable but not loud.

### What likely changes

| Path | Change |
|---|---|
| `src/components/TimerBar.tsx` | Make the running task name clickable; accept new prop. |
| `src/App.tsx` | Wire `onTaskClick` to switch tab + select. |

### Open questions

- If the running task isn't in the user's TaskList (e.g. ran from ClickUp web on a task not assigned to them in Helm's view), `setSelected` lands on a missing id and TaskDetail shows "Loading…" then the not-found state. Need to gracefully handle: probably render TaskDetail by `taskId` even if not in `tasks` (it already fetches by id via `getTaskDetail`). Verify this works.
- Subtle vs. obvious affordance? Lean subtle — single-pixel underline on hover.

## Idea 3 — Day-by-day breakdown on Stats hover

**Today's behavior**: `StatsPanel` shows three flat aggregations (by task / by status / by list) for the week. There's no temporal breakdown — I can't tell whether 4h on a task was Monday-only or spread across 5 days.

**Goal**: Hover a row in the Stats panel → a small tooltip-style overlay shows that row's day-by-day breakdown for the week as horizontal bars. Days with zero work render as zero-width bars (or a faint zero pill) so the day axis is always present, never collapsed.

### Sketch of the approach

**Data shape** — extend the aggregation to also produce per-day buckets per row:

```typescript
interface RowWithDays {
  label: string;
  ms: number;
  color: string | null;
  perDay: number[]; // length 7, Mon=0 ... Sun=6
}
```

For each entry, compute `dayOfWeek = (new Date(e.start).getDay() + 6) % 7` (Mon=0). Bucket into `perDay[dayOfWeek]`.

**Hover UI** — render a small tooltip-card on hover (positioned absolutely, anchored to the cursor or to the row's right edge). Inside the card:
- Stack of 7 mini-bars, one per day, labeled "Mon Tue Wed Thu Fri Sat Sun".
- Each bar: width proportional to that day's hours over the row's max single-day hours.
- Zero days: show the label and an empty bar (zero width with a hairline outline) + the label "0h 00m" on the right.
- Same color as the parent row's bar.

**Implementation**:
- Add `perDay` to the `Row` type in `StatsPanel.tsx`.
- Add a `hoveredRowIndex` state. `Bar` component sets/unsets on `onMouseEnter`/`onMouseLeave`.
- Render the tooltip via React portal or absolutely-positioned div above the panel. CSS `pointer-events: none` so it doesn't flicker on its own hover.
- Keep tooltip within panel bounds — clamp horizontal position.

### Things to watch

- **Performance**: per-day bucketing is one extra `forEach` per entry — negligible (~50 entries/week max).
- **Touch / no-hover devices**: Helm is desktop Electron, so hover always exists. No mobile fallback needed.
- **Tooltip flicker**: anchor the tooltip outside the row's hit area, or use `pointer-events: none` so the cursor stays "on" the row.
- **Day cutoff**: use local-time `startOfDay` (already in `src/utils/time.ts` from 0.3.0).
- **Edge case — zero week**: a row that exists but has no entries this week shouldn't even appear in the list (already filtered). Tooltip never empty.

### What likely changes

| Path | Change |
|---|---|
| `src/components/StatsPanel.tsx` | Add `perDay` to Row, populate during aggregate, add hover state + tooltip render. |
| `src/utils/time.ts` (read-only) | Reuse `startOfDay` and `fmtDuration`. |

### Open questions

- Mon–Fri only or full Mon–Sun? Lean full week — captures weekend work if any.
- Tooltip on hover, or pin-on-click (sticky) for accessibility / keyboard nav? Lean hover for 0.4.1, add click-to-pin in 0.4.x if it feels flaky.
- Show day labels above the bars or inline (left of each bar)? Lean inline-left — more compact, easier to scan.
