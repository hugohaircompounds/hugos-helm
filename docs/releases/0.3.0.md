# Release 0.3.0

Minor release on top of `0.2.1`. Four user-facing features bundled together ‚Äî the high-impact, low-risk subset of the original 0.3.0 plan. The remaining five ideas were carved out into [`0.3.1.md`](./0.3.1.md).

Bumped via `npm version minor` ‚Üí `npm run release` when ready. See [WORKFLOW.md](./WORKFLOW.md) for the full release process.

> **Numbering note**: idea numbers preserved from the original combined plan (3, 4, 5, 9) so cross-references with `docs/releases/0.3.1.md` and the original conversation stay intact. The gaps (1, 2, 6, 7, 8) shipped to 0.3.1.

---

## Why a minor (not a patch)

These four ideas add real new functionality (new IPC channel, new Settings field, new component, new rendering path). That's a `minor` per semver conventions. None are breaking.

The four ideas:
- **Idea 3** ‚Äî Manual time-entry creation (forgot to start the timer? Add it after.)
- **Idea 4** ‚Äî Per-task time-logged badge in `TaskList` (see at-a-glance where my time goes).
- **Idea 5** ‚Äî Pin/favorite tasks (frequently-used tasks stick to the top regardless of status).
- **Idea 9** ‚Äî Make weekly timesheet actually usable (day groups, per-day subtotals, per-day TimelineBars, pagination).

---

## Idea 3 ‚Äî Manual time-entry creation

**Today's behavior**: I can edit existing time entries (added in 0.2.0) and the running timer (added in 0.2.1), but there's no way to *retroactively* add an entry I forgot to start. If I worked 9‚Äì10 on task X without the timer running, that hour is just gone from my timesheet.

**Goal**: A "+ New entry" affordance on the Timesheet that opens a form (task picker + start + end + optional description), creates the entry through ClickUp's `POST /team/{id}/time_entries`, and inserts it into the list and timeline.

### Sketch of the approach

**New IPC**: `createTimeEntry(taskId, start, end, description)` ‚Üí `clickup.createTimeEntry()`. ClickUp's create-entry endpoint (`POST /team/{teamId}/time_entries`) accepts `tid` (task id), `start`, `duration` (in ms), and `description`. Returns the new entry; renderer prepends it to `useTimeEntries.entries`.

**UI**:
- New "+ New entry" button in `TimesheetEditor.tsx` header (next to the range select / Refresh button).
- Form lives in the middle column (replacing/coexisting with `TimeEntryDetail`). Reuse the existing `<input type="datetime-local">` controls from `TimeEntryDetail`.
- **Task picker** ‚Äî autocomplete over current task list. Reuse `tasks` from `useClickUp` already in `App.tsx`. Show task name + list for disambiguation. Allow picking "(no task)" for untracked entries.
- Save ‚Üí IPC ‚Üí optimistic insert.

**Validation**:
- `start < end` (don't allow zero or negative durations).
- Warn (don't block) if the new entry overlaps an existing one ‚Äî the `TimelineBar` will render the overlap stripe naturally.
- `taskId` is optional (untracked entries are valid).

### What likely changes

| Path | Change |
|---|---|
| `electron/services/clickup.ts` | New `createTimeEntry(...)` calling `POST /team/{id}/time_entries`. |
| `electron/ipc/handlers.ts` + `electron/preload.ts` + `shared/types.ts` | New IPC channel `clickup:createTimeEntry`. Extend `HelmApi`. |
| `src/hooks/useTimeEntries.ts` | New `create(...)` callback that calls IPC and prepends to `entries`. |
| `src/components/TimesheetEditor.tsx` | "+ New entry" button + state for "creating mode". |
| `src/components/TimeEntryDetail.tsx` (or new `NewTimeEntryForm.tsx`) | Form for the create flow. Decide whether to reuse `TimeEntryDetail` with a `mode='create'` prop or split into a new component. |

### Open questions

- Reuse `TimeEntryDetail` (one component, two modes) or a separate `NewTimeEntryForm.tsx` (cleaner)? Lean separate ‚Äî different validation rules and a task picker make the shared component messy.
- Should the task picker default to the most-recently-tracked task? Probably yes ‚Äî minor UX win.

## Idea 4 ‚Äî Per-task time-logged badge in `TaskList`

**Today's behavior**: `src/components/TaskList.tsx` shows tasks grouped by status. There's no indication of how much time I've logged on a given task. To see that I have to switch to Timesheet and scan the list.

**Goal**: A small pill on each task row showing total time logged this week (or today, toggleable). Makes "what's eating my time" visible without leaving the Tasks tab.

### Sketch of the approach

**Compute totals in `App.tsx`** so they're shared without re-fetching:
```typescript
const taskTotals = useMemo(() => {
  const map = new Map<string, number>();
  for (const e of timesheet.entries) {
    if (!e.taskId) continue;
    map.set(e.taskId, (map.get(e.taskId) || 0) + (e.duration || 0));
  }
  return map;
}, [timesheet.entries]);
```

Pass `taskTotals` into `TaskList` as a new prop. Render as `data-slot="task-pill"` (already styled in 0.2.0 conventions) using `fmtDuration` from `src/utils/time.ts`.

**Range toggle**:
- Tiny "Today / Week" toggle in `TaskList` header. Drives a separate `useTimeEntries('today' | 'week')` instance, OR we always pull `'week'` and filter to `today` on the renderer side. Probably the latter ‚Äî one less network call.

### What likely changes

| Path | Change |
|---|---|
| `src/App.tsx` | Compute `taskTotals` map; pass to `TaskList`. Maybe lift `useTimeEntries('week')` permanently so totals are always fresh. |
| `src/components/TaskList.tsx` | Accept `taskTotals` prop. Render pill. Tiny range toggle in header. |
| `src/utils/time.ts` | Already has `fmtDuration` ‚Äî reuse. |

### Open questions

- Should the pill show only when `total > 0`, or always (showing `0:00`)? Lean only-when-positive ‚Äî visual noise.
- Color the pill if total exceeds some weekly cap (e.g., > 8h on one task)? Maybe later.

## Idea 5 ‚Äî Pin/favorite tasks

**Today's behavior**: Tasks I context-switch to constantly (e.g. "Daily standup", "Email triage") get buried under whatever status group they happen to be in. No way to pin them to the top.

**Goal**: Star/pin button on each task row. Pinned tasks render in a sticky "Pinned" group at the top of `TaskList` regardless of their underlying status.

### Sketch of the approach

**New `Settings` field**: `pinnedTaskIds: string[]`. Default `[]`.

**Pin button**: Small star icon on each task row in `TaskList.tsx`. Click toggles membership in `pinnedTaskIds`. Debounced save (reuse the `useLayout.ts` debounce pattern).

**Rendering**:
- `TaskList`'s grouping `useMemo` partitions tasks: pinned ones go into a synthetic "Pinned" group at the top; non-pinned tasks group by status as today.
- "Pinned" group respects the same collapse behavior as other groups (key it under `__pinned__` in `collapsedStatusGroups`).
- Order within "Pinned": preserve `pinnedTaskIds` insertion order so the user can curate.

### Things to watch

- A pinned task whose status changes still appears in the Pinned group, not its status group ‚Äî that's the point. Confirm it doesn't render twice (filter pinned out of the per-status groups).
- Interaction with the deferred Idea 2 (status equivalences, in 0.3.1): pinned tasks bypass status grouping entirely, so equivalences won't apply to them. Acceptable.

### What likely changes

| Path | Change |
|---|---|
| `electron/db/index.ts` + `shared/types.ts` | New `pinnedTaskIds: string[]` setting. |
| `src/components/TaskList.tsx` | Star button per row. Pinned partition in grouping `useMemo`. |
| `src/hooks/useLayout.ts` (pattern reused) | Debounced save helper. |

### Open questions

- Pin via star icon vs. right-click menu vs. drag? Star is the obvious answer.
- Cap pin count? Probably not ‚Äî single-user app, self-policing.

## Idea 9 ‚Äî Make weekly timesheet actually usable

**Today's behavior**: `TimesheetEditor.tsx` already has a "Today / This week" range toggle wired through `useTimeEntries('week')` ‚Üí `clickup.listTimeEntries('week')` (`electron/services/clickup.ts:295`). The API call is correct ‚Äî it computes Monday 00:00 ‚Üí now and filters by assignee. So the function "works" in the sense that data is fetched. **But the rendering is unusable for week view**, which is why it feels broken:

1. **Each row only shows time-of-day, not date.** `fmtTime` (`src/utils/time.ts:12`) returns `9:30 AM` style with no day prefix. So Monday's 9:30 entry and today's 9:30 entry look identical. Five entries from yesterday look like a single day's timesheet.
2. **No day separators or per-day subtotals.** Entries are flat-listed sorted by `start desc`. The user can't tell where Monday ends and Tuesday begins.
3. **`TimelineBar` is gated to `range === 'today'`.** In week view, the visual timeline disappears entirely ‚Äî so there's no graphical clue that multiple days are present.
4. **No pagination on `listTimeEntries`.** ClickUp's `/time_entries` paginates around ~100 results by default; for a typical week that's fine, but the function fetches a single page with no `last_page` loop (compare to `getAssignedTasks` which does paginate, `electron/services/clickup.ts:131-140`). A heavy week could silently truncate.

**Goal**: When the range is "This week", the user can clearly see and edit entries from any day of the week. Day boundaries are obvious; subtotals per day are visible; the timeline visualization adapts.

### Sketch of the approach

**Group entries by day in `TimesheetEditor.tsx`**:
- Compute `entriesByDay: { dayStart: number; entries: TimeEntry[]; total: number }[]` in a `useMemo`. Group key = `new Date(e.start).toDateString()` or `startOfDay(e.start)`.
- Sort groups by day descending (most recent first), entries within each group also descending.
- Render each group with a sticky day header:
  ```
  Tuesday, Apr 28 ¬∑ 4h 30m  ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ
  ```
  using `data-slot="panel-header"` styling or a lightweight subheader variant.
- Synthetic running entry from 0.2.1 lands in the most-recent (today) group.

**Today view stays as-is** ‚Äî single ungrouped list, no day headers (would just be one header for one day, noise). Reuse the same grouping function but skip the headers when `range === 'today'`.

**Multi-day `TimelineBar`** ‚Äî going with **Option A**: render one `TimelineBar` per day inside each day's group. Each bar covers 8am‚Äì5pm of that day, mirroring today's behavior. Drops in cleanly without rewriting the bar's coordinate math; just remove the `range === 'today'` gate and pass each day's entries as a slice with a `dayStart` prop.

**Pagination not applicable** ‚Äî early in 0.3.0 build I tried adding a pagination loop to `listTimeEntries` mirroring `getAssignedTasks`. It backfired: the `/team/{id}/time_entries` endpoint does NOT support `page` ‚Äî it returns the entire date range in a single response, and never advertises `last_page`. The loop multiplied every entry ~10√ó until the safety break, inflating day totals and tripping React's duplicate-key warning. Reverted to a single request. Leaving the note here so future-me doesn't re-attempt it.

**Editing entries from previous days** ‚Äî the existing `TimeEntryDetail` middle-column editor already supports any entry by id. With day grouping in place, clicking a Monday row selects it normally and edits work as today. No new edit code needed ‚Äî only the grouping UX. Verify ClickUp's `PUT /time_entries/{id}` accepts edits to entries from earlier in the week (it does ‚Äî there's no time-window restriction).

### Things to watch

- **Day boundaries are local-time, not UTC.** `new Date(ts).toDateString()` uses local time, which is what we want ‚Äî entries that crossed midnight UTC but stayed within the same local day should group together.
- **Cross-midnight entries** (rare ‚Äî e.g. work past midnight): pick a side. Easiest: group by `start`'s local day. If an entry spans midnight, it appears in its start day with a duration that may extend "past" the day's normal range. The TimelineBar's `resolveToday` already handles this for today; the per-day TimelineBars (Option A) inherit the same logic.
- **Synthetic running entry placement**: it has `start = state.startedAt` so its day group is whatever day the timer started ‚Äî usually today. Fine for normal use; weird edge case if a timer ran across midnight. Won't fix in 0.3.0.
- **Scroll position**: when switching from "Today" to "This week", the scroll should land at top (most recent day). Re-key the list on `range` change to force a reset.
- **Performance**: a week of entries is at most ~50 rows; grouping + per-day TimelineBars are negligible. No memoization heroics needed.

### What likely changes

| Path | Change |
|---|---|
| `electron/services/clickup.ts:295` | Add pagination loop to `listTimeEntries` (defensive, matches `getAssignedTasks` pattern). |
| `src/utils/time.ts` | New `startOfDay(ts: number): number` helper (parallel to existing `startOfToday`). New `fmtDayHeader(ts)` that returns e.g. `"Tuesday, Apr 28"`. |
| `src/components/TimesheetEditor.tsx` | Group entries by day in `useMemo`. Render day headers + per-day subtotals when `range === 'week'`. Drop the `range === 'today'` gate around `TimelineBar` (one per day group). |
| `src/components/TimelineBar.tsx` | Accept a `dayStart` prop instead of always using `startOfToday()`, so each instance can render a specific day. |

### Open questions

- Day-header sticky vs. scrolling? Sticky is friendlier on long weeks but adds CSS complexity. Default to non-sticky; revisit if scroll-tracking the current day matters.
- Render empty days (no entries)? Skip them ‚Äî noise. If Tuesday has zero entries, no Tuesday header.
- Show day-of-week as the primary label or the date? Both: `Tuesday, Apr 28`.
- Does Option A (one TimelineBar per day) feel cluttered with 5 stacked bars? If so, fall back to "TimelineBar only on today's group; week's other days are list-only." Decide visually with `npm run dev`.

## Release bundling

Single minor version bump (user-visible new functionality, no breaking changes):

```powershell
git add -A ; git commit -m "feat: manual entry creation, per-task badge, pinned tasks, weekly timesheet UX"
npm version minor
npm run release
git push --follow-tags
```

Installed Helm picks up `0.3.0` on next launch. See [WORKFLOW.md](./WORKFLOW.md) for the full workflow reference.
