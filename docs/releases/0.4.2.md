# Release 0.4.2

Patch on top of `0.4.1`. Four themes — a one-line version-in-titlebar fix, a deep pass on the task-comments section that takes it from read-only to a full interaction surface (including @mentions), the ability to create a new task in any list on any space, and **a text search box at the top of the Tasks tab** for finding tasks by name. Decisions on the open questions are locked in below; **2d (👍 reactions) is dropped** because ClickUp's public REST API has no task-comment reaction endpoint. **Not ready to build**, but ready to scope.

Bumped via `npm version patch` → `npm run release` when ready. See [WORKFLOW.md](./WORKFLOW.md) for the full release process.

---

## Why a patch (not minor)

**Idea 1** is a one-liner. **Idea 2** adds four new IPC methods (post comment, post reply, load thread replies, list workspace members), reshapes the `Comment` type, and switches the comment write path from plaintext to ClickUp's structured `comment` array format so @mentions actually fire notifications. **Idea 3** adds four more IPC methods (list folders, list folderless lists, list lists in folder, create task) and one new modal. **Idea 4** is renderer-only — pure UI, no new IPC. All additive — no removed surface, no migrated fields, no new tab, no new scheduler job — so semver-wise it's patch-shaped.

**In effort terms this release is firmly 0.5.0 territory** (~6.5–7.5 days of work total). Calling it 0.4.2 is the user-directed choice and matches the 0.4.x "small targeted improvements" cadence in spirit; in practice this is the largest single release since 0.3.0. Three more lines of justification:

- All four ideas are in scope per explicit user direction ("all of these features must be included in the next patch").
- No breaking changes to any existing IPC, type, or UI behavior, so semver hasn't been violated.
- The auto-update path (`electron-updater`) makes patch-vs-minor a UX label, not a delivery mechanism — users just see "Update ready" either way.

The four items:
- **Idea 1** — Show the running version in the window title bar so it's discoverable without opening Settings or DevTools.
- **Idea 2** — Take the task-comment section from read-only to a real interaction surface: load all comments (paginated), group replies under their parent thread, post new top-level + thread replies from inside Helm, and **@mention teammates with real notifications**. **Reactions (2d) are blocked on ClickUp's public API** — see resolution below.
- **Idea 3** — Create new tasks in any list on any space, from a "+ New task" affordance in the Tasks tab, without leaving Helm.
- **Idea 4** — Search the Tasks tab by name (with list-name as a secondary match), composing with the existing chip filters.

## Idea 1 — Show version in window title

**Today's behavior**: `electron/main.ts:25` sets `BrowserWindow.title = 'Helm'`, but `index.html:6` has `<title>Helm</title>`, which overrides the BrowserWindow value as soon as the page loads. The version isn't visible anywhere in the running app.

**Goal**: The OS window title reads `Helm 0.4.2` (or whatever the current version is). Same value is queryable from DevTools console as `await window.helm.getAppVersion()` so renderer code can render it elsewhere if we ever want to.

### Sketch of the approach

Two surfaces to update because the renderer's `<title>` wins by default:

1. **Add an IPC for the version** — small but earns its keep:
   - `shared/types.ts` → add `getAppVersion(): Promise<string>` to `HelmApi`.
   - `electron/preload.ts` → wire `ipcRenderer.invoke('helm:get-app-version')`.
   - `electron/ipc/handlers.ts` → `ipcMain.handle('helm:get-app-version', () => app.getVersion())`.

2. **Set the title from the renderer once on mount** — in `src/App.tsx`, on first render: `window.helm.getAppVersion().then((v) => { document.title = \`Helm ${v}\`; })`. This is the only path that beats the static `<title>` in `index.html`.

Don't bother changing `BrowserWindow.title` in main — it's masked by the HTML `<title>` after load, so making it accurate there is dead code unless we also intercept `page-title-updated`. Renderer-side is simpler.

### Things to watch

- **Pre-load flicker**: between `BrowserWindow` opening and the renderer's effect running, the title shows `Helm`. Acceptable — it's <500ms in practice. If we want to avoid it, listen for `page-title-updated` in main and rewrite, but that's overkill for this.
- **Auto-update lifecycle**: when an update is downloaded, the *running* app is still on the old version until restart. The title should reflect the running version, which is what `app.getVersion()` returns. Confirmed correct.

### What likely changes

| Path | Change |
|---|---|
| `shared/types.ts` | Add `getAppVersion(): Promise<string>` to `HelmApi`. |
| `electron/preload.ts` | Wire `ipcRenderer.invoke('helm:get-app-version')`. |
| `electron/ipc/handlers.ts` | Register `ipcMain.handle('helm:get-app-version', ...)` returning `app.getVersion()`. |
| `src/App.tsx` | One-shot effect on mount: read version, set `document.title`. |

## Idea 2 — Refine the comment section under each task

The comment section in `TaskDetail.tsx` is currently read-only, paginated to ClickUp's default page size (~25), and renders comments as a flat list with no awareness of replies. This idea promotes it to a real interaction surface.

### 2a — Load all comments

**Today's behavior** (`electron/services/clickup.ts:232-258`): `getTaskDetail` calls `GET /task/{id}/comment` once. ClickUp's response is capped (default page size; oldest comments are dropped on long threads). On a task with 60 comments, only ~25 surface — the user thinks history is gone.

**Goal**: All comments load, regardless of count. Visual: a flat list ordered chronologically; oldest at top or newest at top — pick one and stay consistent (lean **newest-first** to match the existing render order in `TaskDetail.tsx:280`).

#### Sketch of the approach

ClickUp paginates the comment endpoint via a `start` cursor + `start_id` (the timestamp + id of the boundary comment). Loop until the response yields an empty `comments` array:

```typescript
// Pseudocode in clickup.ts
async function fetchAllComments(taskId: string): Promise<RawComment[]> {
  const out: RawComment[] = [];
  let start: number | undefined; // unix ms
  let startId: string | undefined;
  while (true) {
    const params = new URLSearchParams();
    if (start) params.set('start', String(start));
    if (startId) params.set('start_id', startId);
    const resp = await cu<CommentsResp>(
      `/task/${encoded}/comment?${params}`
    ).catch(() => ({ comments: [] }));
    if (!resp.comments.length) break;
    out.push(...resp.comments);
    const oldest = resp.comments[resp.comments.length - 1];
    start = Number(oldest.date);
    startId = oldest.id;
    if (resp.comments.length < 25) break; // last page
  }
  return out;
}
```

Bound the loop (max ~10 pages = 250 comments) so a runaway never blocks `getTaskDetail`. Log + break above the cap.

### 2b — Organize comments by thread

**Today's behavior**: every reply renders as a sibling of the parent — context for "who replied to what" is lost.

**Goal**: Top-level comments render at the section root; replies render indented under their parent.

**Resolved load strategy** (questions 1+2): the most recent thread is auto-expanded with its replies fetched eagerly; every other thread starts collapsed and lazy-loads its replies on first expand. "Most recent" is defined by **most recent activity** — `max(comment.dateCreated, max(reply.dateCreated))` — so a 6-month-old top-level comment that just got a fresh reply correctly bubbles up as the auto-expanded one.

#### Sketch of the approach

ClickUp's data model: each top-level `Comment` has a `reply_count` (number of replies). Replies live behind `GET /comment/{commentId}/reply` (separate endpoint, returns `{ comments: [...] }` shaped same as top-level).

Two-pass load inside `getTaskDetail`:

1. **Pass 1** — `fetchAllComments(taskId)` returns top-level comments + their `reply_count`. To find the most-recent-active thread without fetching every reply, ClickUp's top-level response includes a `date_updated` field (timestamp of latest activity in that thread). Sort by that; pick the head.
2. **Pass 2** — for the head thread only, fire `GET /comment/{headId}/reply`. Store on `head.replies`. Every other top-level gets `replies: []` + a flag indicating "not loaded yet."

When the user clicks "Show N replies" on a collapsed thread, the renderer calls a new IPC `loadCommentReplies(commentId)` that hits the same endpoint and merges the result into local state + cache. After first expand, replies are cached for the session — re-collapsing and re-expanding doesn't refetch.

Extend the `Comment` type:

```typescript
// shared/types.ts
export interface Comment {
  id: string;
  text: string;
  user: string;
  dateCreated: number;
  dateUpdated: number;          // NEW — used to pick the most-recent thread
  replyCount: number;           // NEW — informs the "Show N replies" affordance
  parentId: string | null;      // NEW — null for top-level
  replies: Comment[];           // NEW — populated only after explicit fetch
  repliesLoaded: boolean;       // NEW — false on top-level until expand-fetch lands
}
```

Render: `TaskDetail.tsx`'s comment `<ul>` becomes a nested structure. Replies are visually offset (`pl-6` or similar) and toggled by a small "▸ N replies" / "▾ N replies" disclosure. The head thread renders pre-expanded with its replies already populated; every other thread renders collapsed even if `replyCount > 0`. First click on a collapsed disclosure shows a brief "Loading…" then the replies; subsequent clicks just toggle visibility.

#### Things to watch

- **`date_updated` reliability** — needs verification. ClickUp's top-level comment response includes a `reply_count`, but I'm not 100% sure it also includes `date_updated` reflecting reply activity. If it doesn't, fall back to: head = most recent top-level by `dateCreated`. That's the more conservative pick and still matches the spirit of the rule (recent activity floats up). Verify with one curl before building.
- **Renderer cache invalidation** — `useTaskDetailCache` already replaces the cache entry on save. Posting a new comment / reply or expanding-and-loading replies needs to do the same: `cache.set(detail.id, nextDetail)`. Otherwise stale comments survive a tab switch.
- **Cold-load latency** — pass 1 + pass 2 in parallel with the existing task fetch. Net wire time should match today's load (one extra `/reply` request) since we already fire two requests via `Promise.all`. Add the head-thread `/reply` request to the same `Promise.all`.

### 2c — Respond to comments from inside Helm

**Today's behavior**: read-only — no compose box.

**Goal**: Two visually distinct affordances so the user always knows which type of post they're sending:
- **New thread** — a single textarea at the bottom of the comments section, labeled `Start a new thread` (placeholder: "What's on your mind? This starts a new thread on this task."). Posts to `POST /task/{id}/comment`.
- **Reply in existing thread** — a "Reply" button on each thread head opens an inline textarea **inside** that thread's nested area, labeled `Reply to {author}'s thread · {date}` (e.g. `Reply to Hugo's thread · Apr 24`). Posts to `POST /comment/{threadId}/reply`. Submitting collapses the inline composer and appends the new reply to that thread's `replies[]`.

The visual treatment is the differentiator — same submit button copy is fine, but the **container** for "new thread" is the section-bottom card, and the **container** for "reply" is the indented thread block with the parent's author/date label visible above the textarea. No way to confuse them.

#### Thread tracking through the API

ClickUp doesn't model "thread" as a first-class entity — each top-level comment's `id` *is* the thread identifier:

- `POST /task/{taskId}/comment` → returns a new top-level comment with id `T`. That id `T` is the thread.
- `POST /comment/T/reply` → posts into thread `T`. Reply gets its own id `R` but `R.parentId === T`.
- `GET /comment/T/reply` → returns all of thread `T`'s replies. Refetch any time to refresh just one thread.
- ClickUp does **not** allow reply-to-reply (no deep nesting). All replies hang flat under the top-level. So the data structure is two-level: top-level → flat `replies[]`. Helm's `Comment.replies: Comment[]` matches this shape exactly; we don't need a recursive tree.

This means Helm can do everything thread-related the API permits: open a specific thread, watch its activity, post into it, refresh just it. The only thing the public API doesn't expose is per-thread subscription state (you can't subscribe to *thread T's notifications only*; subscriptions are at the task level via watchers).

#### Sketch of the approach

Two new IPC methods, distinct names mirroring the API split so call sites read naturally:

```typescript
// shared/types.ts on HelmApi
// Starts a new thread on a task. Returns the new top-level comment.
createTaskComment(taskId: string, text: string, notifyAll: boolean): Promise<Comment>;
// Posts into an existing thread. parentCommentId is the thread id.
createCommentReply(parentCommentId: string, text: string, notifyAll: boolean): Promise<Comment>;
```

ClickUp endpoints:
- Top-level (new thread): `POST /task/{id}/comment` with `{ comment_text, notify_all, assignee: null }`.
- Reply (existing thread): `POST /comment/{threadId}/reply` with `{ comment_text, notify_all }`.

Both responses include the new comment's id + date. Normalize via the same shape as `getTaskDetail` so the merge into local state is uniform. The reply normalizer sets `parentId = threadId`; the top-level normalizer sets `parentId = null`.

After a successful post:
- **New thread**: insert at the top of `detail.comments` (newest-first); auto-expand it (it becomes the new "head" thread).
- **Reply in thread T**: append to `comments.find(c => c.id === T).replies`; ensure thread T is expanded so the user immediately sees their reply land.
- `cache.set(detail.id, newDetail)` so the next navigation away/back doesn't drop the just-posted comment.
- Clear the appropriate textarea, refocus.

#### `notify_all` — what it actually does, and the chosen default

ClickUp's comment-create endpoints accept a `notify_all: boolean` flag. The behavior:

- **`true`** — every person attached to the task (assignees, watchers, the comment thread's prior participants) gets an in-app notification, **and** the email-notification ones in that set get an email. The same flag governs replies (`POST /comment/{id}/reply`).
- **`false`** — the comment is created and visible in the thread, but **no one is pinged**. The thread itself updates in the ClickUp UI for anyone who happens to be looking, but no notification is dispatched.

The asymmetry to remember: `notify_all: false` is **not** "notify nobody" in the strictest sense — anyone subscribed to ClickUp's "all comments" digest, or anyone with the task open in real time, still sees the comment. It's specifically the *push notification* and *email* paths that are gated.

**Default for Helm**: `notify_all: false`. Reasoning:

- The most common Helm comment is a quick self-note while time-tracking ("started this on the staging branch", "blocked on Hugo"). Pinging everyone for those is noise.
- The riskier failure mode is "I accidentally emailed five people" rather than "I posted a quiet comment that needed louder broadcast." Easier to recover from the latter (re-comment with notify on, or @ someone explicitly) than the former.
- The ClickUp web UI defaults to `notify_all: true` with a visible toggle. Helm intentionally inverts the default to match Helm's role as a personal time-tracking surface, not a team comms surface.

**UI surface for the toggle**: a small `[ ] Notify everyone watching this task` checkbox under the compose textarea, **unchecked by default**. State doesn't persist across sessions — fresh post = fresh decision. If the user posts the same comment twice in a row and forgets to check it the second time, that's by design (default off > sticky-on after one true post). Use the longer copy ("everyone watching this task") instead of "Notify watchers" so it's unambiguous that this is broadcast, not "notify whoever I @-mentioned."

#### Why we use the structured `comment` array (not `comment_text`)

ClickUp accepts comments two ways. The choice is forced by the @mention requirement (see 2e):

- **`comment_text`** (plaintext) — anything typed including `@hugo` is stored as literal text. Mentions are **not registered**, no notification fires. **Not used** in 0.4.2.
- **`comment` array** (structured) — required for real @mentions: `{ comment: [{ text: "..." }, { type: "tag", user: { id: 1234567 } }, { text: "..." }] }`. **This is what 0.4.2 sends.** Plain segments are wrapped as `{ text: "..." }`; resolved mentions become `{ type: "tag", user: { id } }`.

**Implication for `notify_all`**: with the structured array path, both notification mechanisms work — the checkbox handles broadcast, and @-mentions notify their specific targets regardless of the checkbox state. Users get the ClickUp-web mental model (typing `@hugo` pings Hugo) without needing the broadcast toggle for that case.

The conversion from typed text → structured array happens in the renderer just before posting; main process receives the structured payload directly. See 2e for the autocomplete + resolution flow.

#### Other things to watch (2c)

- **Markdown vs. plaintext** — beyond mentions, the same `comment_text` constraint blocks rich formatting (bold, code blocks, links). Acceptable for 0.4.2.
- **Optimistic vs. await-then-render** — lean **await-then-render** for first cut. Posting a comment is rare enough that 200ms of "Saving…" is acceptable, and an optimistic insert that fails leaves the user wondering if it sent.

### 2d — React with 👍 — DROPPED from 0.4.2

**Verification result**: ClickUp's documented public REST API has **no endpoint for adding/removing reactions on a task comment**. The reaction surface that does exist is for **chat messages** (`POST /v3/workspaces/{wid}/chat/messages/{msg_id}/reactions` and the matching `DELETE`). Task comments and chat messages are different entities with different ids, so the chat endpoint isn't a substitute.

The reaction feature visible in ClickUp's web UI on task comments is implemented through their internal API. Calling that from Helm is technically possible (sniff DevTools, replay the request) but:
- Undocumented → can break without warning, and we'd find out via silent regression.
- Personal API tokens may not be authorized against private endpoints; the web UI uses session cookies + CSRF tokens that we don't have.
- Crosses the "only depend on documented surface" line that the rest of the ClickUp integration follows.

**Decision**: 2d is dropped from 0.4.2 entirely. Re-evaluate when ClickUp publishes a public reaction endpoint for task comments. Option to revisit:
- File a request on https://feedback.clickup.com/public-api/ asking for `POST /comment/{id}/reaction` parity with chat. Cheap to do, low odds of fast turnaround.
- Watch the changelog at https://developer.clickup.com/changelog for `comment` + `reaction` keywords.

### 2e — @mention support

**Today's behavior**: no autocomplete, no mention resolution. With 2c-as-originally-planned, typing `@hugo` would render as plain text and Hugo wouldn't be notified.

**Goal**: Type `@` in any compose textarea (new-thread or reply) → an inline dropdown appears showing workspace members filtered by what's typed after the `@`. Pick one (arrow keys + Enter, or click) → the typed `@hugo` is replaced with a styled "mention chip" that the renderer treats as a tag. On Post, the textarea contents are converted to ClickUp's structured `comment` array, with chips becoming `{ type: "tag", user: { id } }` segments. Hugo gets a real ClickUp notification.

#### Sketch of the approach

**Workspace member fetch** — new IPC + ClickUp endpoint:

```typescript
// shared/types.ts on HelmApi
listWorkspaceMembers(): Promise<WorkspaceMember[]>;

export interface WorkspaceMember {
  id: number;        // ClickUp user id (number, not string)
  username: string;  // @-name, no leading @
  email: string;
  initials: string;  // for avatar fallback
  color: string | null;
  profilePicture: string | null;
}
```

ClickUp endpoint: `GET /team/{teamId}/member` returns `{ members: [{ user: {...} }, ...] }`. `ensureWorkspace()` already gives us `teamId`. Result is small (typically <200 members for a typical workspace) and stable, so cache it in main for the session — no need to refetch on every `@` keystroke. Invalidate on `Settings: refresh` or after 10 minutes, whichever first.

**Renderer compose component** — replace the current `<textarea>` with a small compose component that:

1. Tracks the cursor position and watches for `@` triggers.
2. When `@` is typed (or text after `@` changes), opens a positioned dropdown anchored just below the cursor with a fuzzy-filtered member list (filter by username + initials + email-prefix).
3. Arrow keys navigate, Enter selects, Escape closes. Mouse click also selects.
4. On selection, the typed `@hugo` token (from the `@` to current cursor) is replaced with a span styled as a chip: `<span data-mention-id="123" class="mention">Hugo</span>`. The text node is broken so the chip is atomic — backspace deletes it whole.
5. Internal model is a sequence of segments: `{ kind: 'text', value: string } | { kind: 'mention', userId: number, display: string }`. The textarea is *visually* a contenteditable region rendering this model; the model is the source of truth.

**Conversion to structured `comment` array** on Post:

```typescript
// pseudocode
function segmentsToClickUpComment(segments: Segment[]): ClickUpCommentSegment[] {
  return segments.map((s) =>
    s.kind === 'text'
      ? { text: s.value }
      : { type: 'tag', user: { id: s.userId } }
  );
}
```

Adjacent text segments are coalesced. Empty leading/trailing text segments are dropped. Result is the body for `POST /task/{id}/comment` or `POST /comment/{threadId}/reply`.

**Render side (existing comments)** — ClickUp's `GET /task/{id}/comment` response includes both `comment_text` (the rendered plaintext, mentions inlined as `@username`) AND `comment` (the structured array with tag objects). For 0.4.2, render via the structured array so mentions can be styled as chips matching the compose UI. If `comment` is missing in the response (rare; mostly old comments or edge cases), fall back to `comment_text` rendered plain.

#### Things to watch

- **Contenteditable is gnarly**. A textarea cannot host inline atomic chips — it's plain text only. So the compose surface has to be a `contenteditable` div, which has well-known browser quirks (paste handling, selection-on-mention-boundaries, IME composition). For 0.4.2 first cut: **handle the common case cleanly** (typing, arrow keys, backspace deleting a chip whole, paste-as-plaintext). Skip rich formatting, RTL text, and IME edge cases — note them as 0.4.3+ polish.
- **Trigger character detection**. Naive "any `@` opens dropdown" is wrong — `email@domain.com` shouldn't trigger. Rule: open the dropdown only when `@` is preceded by start-of-line, whitespace, or punctuation. Close when the user types whitespace without picking, or moves the cursor away.
- **Member fetch failure**. If `listWorkspaceMembers()` fails (offline, 401, ClickUp 502), the `@` trigger should still type a literal `@` — *not* break the textarea. Show a small inline note: "Mentions unavailable — list failed to load" near the dropdown anchor for that compose. Don't block posting.
- **Mention chip display name vs. real username**. ClickUp's `username` is the canonical handle. Display the canonical name in the chip (so what the user sees in the chip matches what teammates will see in the rendered comment). If `username` is empty (it can be, for some accounts), fall back to `email.split('@')[0]`.
- **Self-mention**. ClickUp lets you @-mention yourself. Allow it — it's how some users mark "for my own follow-up." No special handling.
- **Chip in the API response**. After posting, ClickUp returns the new comment with a `comment` array including the tag objects. The renderer's existing render path (above) handles this uniformly with already-loaded comments. No special "just-posted" rendering needed.

## Idea 3 — Create a new task in any list on any space

**Today's behavior**: The Tasks tab is a read view of tasks assigned to the user. There's no path to create a new task from inside Helm — the user has to switch to ClickUp web, create the task, and wait for the next refresh (or a remote-triggered sync) for it to appear in Helm.

**Goal**: A `+ New task` button in the Tasks tab header. Clicking it opens a compact modal that lets the user pick a target list (cascade: space → folder → list, with "folderless lists" handled), fill task details, and create. After create, Helm refreshes the assigned-tasks view; if the new task is assigned to the user, it appears immediately. Otherwise, a small toast confirms the create with the task's URL ("Created — Open in ClickUp").

### Sketch of the approach

#### ClickUp's space/folder/list model (shape recap)

A workspace contains spaces. A space contains folders **and** "folderless lists" (lists that live directly in the space, not under a folder). A folder contains lists. Tasks live in lists. So the picker is a three-step cascade:

```
Workspace
└── Space (e.g. "Engineering")
    ├── Folder (e.g. "Frontend")
    │   └── List (e.g. "Bugs")  ← tasks created here
    │       Task
    └── (folderless List — direct child of Space)
        Task
```

Helm already has `listSpaces()`. The new IPC adds the rest:

```typescript
// shared/types.ts on HelmApi
listFolders(spaceId: string): Promise<ClickUpFolder[]>;
listListsInFolder(folderId: string): Promise<ClickUpList[]>;
listFolderlessLists(spaceId: string): Promise<ClickUpList[]>;
createTask(listId: string, payload: NewTaskPayload): Promise<Task>;

export interface ClickUpFolder {
  id: string;
  name: string;
}

export interface ClickUpList {
  id: string;
  name: string;
  // Whether this list lives directly under the space (no folder).
  // Pure metadata for the picker; ClickUp's create-task endpoint doesn't care.
  folderless: boolean;
}

export interface NewTaskPayload {
  name: string;                    // required
  description?: string;            // markdown plaintext
  status?: string;                 // must match a status in the list
  priority?: 1 | 2 | 3 | 4 | null; // ClickUp's priority enum
  dueDate?: number | null;         // unix ms
  assignees?: number[];            // ClickUp user ids
}
```

ClickUp endpoints used:
- `GET /space/{spaceId}/folder?archived=false` → folders.
- `GET /folder/{folderId}/list?archived=false` → lists in folder.
- `GET /space/{spaceId}/list?archived=false` → folderless lists in space.
- `POST /list/{listId}/task` with `{ name, description, status, priority, due_date, assignees }`.

  [_______________________________________________]

  [Open ▾]          [Normal ▾]      [datetime-local]

Behaviors:
- **Cascade**: changing Space invalidates Folder + List + Status. Changing Folder invalidates List + Status. Changing List populates Status dropdown via existing `useListStatuses(listId)` hook.
- **Initial state**: the picker pre-selects the space/folder/list of the **currently-selected task** in TaskList (if any). Otherwise, last-used list (persisted in `settings` table — see "What likely changes" for the new key). Otherwise, no selection — user picks from scratch.
- **`+ me` button** under Assignees: instantly adds the current user to the assignee list. Useful since the most common case is "create a task assigned to me." Uses the cached "me" id from 2b/2e's workspace-member fetch (free reuse).
- **Name validation**: trim + nonempty. Empty disables the Create button. Long names (ClickUp soft-limits at ~1024 chars) trigger inline error before submit.
- **Submit**: calls `createTask(listId, payload)`. On success: close modal, optimistically prepend to TaskList if assigned-to-me, fire `useTasks().load()` to canonicalize. On failure: keep modal open, show inline error, preserve all field values.

#### Things to watch

- **Picker tree caching**. Spaces/folders/lists are stable on the timescale of a session. Cache them in main with a 10-minute TTL (matches the workspace-members cache pattern from 2e). Refetch eagerly when the modal opens but serve the cached version while the network request flies. No spinner unless cache is empty.
- **Status dropdown is list-dependent**. ClickUp validates `status` against the chosen list's configured statuses. The existing `getListStatuses(listId)` already gates this for TaskDetail; reuse here. Default to the list's first non-closed status (typically "Open" or "To Do").
- **Folder + list dropdown ergonomics**. For a workspace with N folders × M lists each, the cascade can feel clunky. Cheap quality-of-life: type-to-filter on each dropdown. Out of scope if it's any work; in scope if the existing pattern from `StatusSelect` extends easily.
- **No subtasks in v1**. ClickUp distinguishes top-level tasks from subtasks (`parent` field on the create payload). Subtasks would require selecting a parent task, which adds another picker step. Out of scope for 0.4.2 — top-level tasks only. Future minor.
- **Creating a task assigned to someone else**: `assignees` accepts numeric user ids. The picker shows the same workspace-member list 2e fetches. Multi-select. If the user creates a task and isn't an assignee, the task will not appear in their assigned-tasks list — handle this gracefully with a toast: `Created in {ListName}. Open in ClickUp →` (link uses the response's `url` field).
- **Currently-running timer**: irrelevant. Creating a task does not start a timer, does not interrupt one. The new task simply exists; timing it is a separate user action.
- **Workspace tier limits**. ClickUp's free tier caps custom statuses, custom fields, etc. — but `POST /list/{id}/task` itself works on every tier. Custom-field setters are out of scope; only the basic fields above are settable.

## Idea 4 — Search the Tasks tab

**Today's behavior**: `TaskList.tsx` has chip-based filters (status, list, priority, due date) via `TaskFilterBar.tsx`, and lets you toggle/persist combinations. There's no text search — finding a specific task by name means scrolling, eyeballing, or remembering which list/status filter narrows it best.

**Goal**: A search input at the top of the Tasks tab (above the existing Filters bar). Type → the visible task list narrows to matches in real time. Search composes with the existing chip filters: a task is shown if it matches **the search** AND **all active filters**. Empty query = filters-only behavior (today's flow, unchanged).

### Sketch of the approach

#### Match semantics

- **Fields searched**: task name (primary) + list name (secondary). Two fields keeps the surface unambiguous; description is unavailable in the list-row data, status/priority are already filterable as chips.
- **Case insensitive.**
- **Token-AND substring**: split the query on whitespace, every token must appear as a substring in *either* `name` or `listName` (any field, not all). E.g. `wire ui` matches a task `Wire up the new UI` in list `Frontend`, and matches a task `Wire harness` in list `UI Backlog`.
- **No fuzzy matching in v1** — substring is enough for ~500 in-memory tasks and avoids the "why did this match?" surprise. If it feels imprecise after a week, escalate to a fuzzy library (e.g. `fuse.js`) in 0.4.3.

#### UI placement

A thin row above the existing `<TaskFilterBar>`:

```
┌──────────────────────────────────────────────────────┐
│  🔍  [ search tasks…________________________ ]   [×] │  ← TaskSearchBar
├──────────────────────────────────────────────────────┤
│  ▸ Filters · 2          Clear                        │  ← TaskFilterBar (today)
├──────────────────────────────────────────────────────┤
│  ⌫ Status: To Do                                     │
│  ▶ task A                                            │
│  ▶ task B                                            │
└──────────────────────────────────────────────────────┘
```

- 🔍 icon (or just placeholder text) to telegraph purpose.
- `×` clear button appears only when query is non-empty.
- Press `Escape` while focused → clear query, keep focus.
- Press `/` while not in any other input → focus the search input. Cheap discoverability for keyboard users; `Cmd+F` is left alone (browser-find is useful within Helm too).
- The Filters bar stays unchanged — search is its own thing, not a chip.

#### State + persistence

```typescript
// TaskList.tsx state
const [query, setQuery] = useState('');           // not persisted
const [filters, setFilters] = useState<TaskFiltersState>(EMPTY_FILTERS); // already persisted
```

- **Not persisted to disk.** Fresh session starts with an empty query — surfacing a hidden narrowing on launch ("why are tasks missing?") is a worse failure mode than asking the user to re-type.
- **Survives tab switches in-session.** `query` is component state on `TaskList`; switching to Timesheet and back doesn't unmount — preserved automatically.
- No new IPC, no new settings KV.

#### Filtering integration

`TaskList.tsx`'s existing `passesFilters(t, filters)` stays as-is. Add a peer:

```typescript
function passesSearch(t: Task, query: string): boolean {
  const q = query.trim().toLowerCase();
  if (!q) return true;
  const tokens = q.split(/\s+/);
  const hay = `${t.name ?? ''}\n${t.listName ?? ''}`.toLowerCase();
  return tokens.every((tok) => hay.includes(tok));
}
```

`useMemo` for `filtered`:

```typescript
const filtered = useMemo(
  () => tasks.filter((t) => passesFilters(t, filters) && passesSearch(t, query)),
  [tasks, filters, query]
);
```

Pinned tasks already flow through `filtered` today — same rule applies: a pinned task that doesn't match the search isn't rendered. Consistent with how chip filters already work.

#### Empty-results state

When `filtered.length === 0` AND `query !== ''`:
- Render an empty-state block in the task list area: `No tasks match "{query}"` with a `Clear search` button (clears query only, leaves filters).
- If filters are also active, show a secondary line: `2 active filters may also be hiding results — clear filters?` with a button. Helps disambiguate "is it the search or the filter that's hiding everything?"

When `filtered.length === 0` AND `query === ''` (filters only): keep the existing today behavior unchanged.

#### Header count

Today: `{filteredCount}` of `{total}`. After search: same math, since `filtered` already accounts for both passes. No copy change needed — the count is honest about what's visible regardless of which mechanism narrowed it.

### Things to watch

- **Debounce**: not needed for typing into a 500-element JS filter — re-running on every keystroke is sub-millisecond. Don't add complexity for a non-problem.
- **Highlighting matches in task rows**: out of scope for v1. Visual highlighting of matched substrings is a polish item and adds rendering complexity. The list itself narrowing to results is sufficient signal.
- **Search applied across pinned section**: pinned tasks must match the search to appear. If the user pins "task A" and searches for "task B", the pin section just doesn't render. Consistent with how filter chips already affect pins.
- **`/` hotkey conflicts**: only fire when no input/textarea/contenteditable has focus, and not while a modifier key is held. Standard pattern; small effect handler in TaskList.
- **Overlay with Idea 3**: when the Create-Task modal is open, the `/` hotkey should not steal focus. The modal owns focus — listener checks `document.activeElement` lineage.
- **Internationalization**: `toLowerCase()` is locale-bound; for English/Western text it's fine. If we ever support non-English task names, switch to `localeCompare` semantics. Not a problem today.

### What likely changes (Idea 4)

| Path | Change |
|---|---|
| `src/components/TaskSearchBar.tsx` (new) | Small input + clear button + `/` hotkey listener. ~40 LOC. |
| `src/components/TaskList.tsx` | Add `query` state, `passesSearch`, mount `<TaskSearchBar>` above `<TaskFilterBar>`. Wire empty-results state when `query` is non-empty and `filtered.length === 0`. Update the `filtered` memo deps. |
| `shared/types.ts` | No change. Search is not persisted to settings, so no new fields. |
| `electron/*` | No change. Renderer-only feature. |
