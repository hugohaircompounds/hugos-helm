## Why a patch (not 0.3.0)

These are small refinements to the timesheet UX shipped in 0.2.0. No new surfaces, no breaking changes, no new IPC contract — purely renderer-side polish that makes the running timer visible in places it currently isn't.

## Idea 1 — Show the currently-running task inside the timesheet

**Today's behavior**: The timer bar at the top shows the running task. The Timesheet list and the `TimelineBar` only show entries returned by `clickup.listTimeEntries()`, which hits `GET /team/{id}/time_entries` — that endpoint does **not** include the in-progress entry. So the timer is visible in the header but invisible in the timesheet panel until it's stopped.

**Goal**: The active timer should appear in three places simultaneously, all updating live:
1. The header `TimerBar` (already works — keep as-is).
2. As a row in the timesheet list (`TimesheetEditor`), at the top.
3. As a live-growing segment on the `TimelineBar` (the bar already supports `running` segments via `entry.end === null`; just needs the entry in the array).

### Sketch of the approach

The cheapest path is **synthetic merge in the renderer**, not new IPC:

- `useTimeEntries` (or a small wrapper hook above it) accepts the current `TimerState` and, when `state.running`, prepends a synthetic `TimeEntry` to the list it returns:
  ```typescript
  // Pseudocode
  const runningEntry: TimeEntry = {
    id: `__running:${state.taskId}`, // sentinel id, never sent to ClickUp
    taskId: state.taskId,
    taskName: state.taskName,
    description: '',                  // running entries have no description yet
    start: state.startedAt,
    end: null,                        // signals "running" to TimelineBar
    duration: Date.now() - state.startedAt,
  };
  ```
- The list renders the row with a pulsing dot / "Running" badge in place of the duration. Click → middle column shows a read-only `TimeEntryDetail` (or just a placeholder "Stop the timer to edit") since ClickUp's PUT on a running entry has historically been flaky.
- `TimelineBar` already renders `running` segments with a pulsing right edge — it just needs this entry in its `entries` prop.
- A 1s tick (or the existing `useTimer` elapsed tick) drives the row's live duration display. **Do not** introduce a new clock — reuse whatever drives the header.
- On `timer:stopped` event from `timerBus`, kick a `load()` of `useTimeEntries` so the real persisted entry replaces the synthetic one. Guard against the brief window where both could exist (filter by id-prefix `__running:` when the real entry's `taskId` matches and its `start` is within ~5s).

### What likely changes

| Path | Change |
|---|---|
| `src/hooks/useTimeEntries.ts` | Accept `timerState` (or wrap with a parent hook); merge synthetic running entry. |
| `src/components/TimesheetEditor.tsx` | Render the synthetic row distinctly (pulsing indicator, no Delete button). |
| `src/components/TimeEntryDetail.tsx` | Treat synthetic ids as read-only / show a "stop to edit" hint. |
| `src/App.tsx` | Pass timer state into the hook (or compose at the caller). |
| `src/components/TimelineBar.tsx` | Likely no change — already handles `end === null`. Verify visually. |

### Out of scope for 0.2.1

- Editing the running entry's description before stopping (keep using the EOD prompt flow).
- Cross-day running entries (rare; clamping in `TimelineBar.resolveToday` already handles the visual side).
- Server-side change to `listTimeEntries` to include the running one (would require backend work and adds nothing the renderer can't do itself).

## Idea 2 — Desktop notification at 4:55 PM that opens the Timesheet editor (no autofill)

**Today's behavior**: At 16:55 the `eod-prompt` job (`electron/scheduler/jobs.ts:114`) does two things:
1. Emits a `description-prompt` event on `timerBus` with `kind: 'eod'` and **`defaultText: titles.join('\n')`** — the renderer's `DescriptionPrompt.tsx` modal opens **pre-filled** with today's task titles concatenated.
2. Logs to `job_logs`.

If Helm is in the background, the modal is invisible until I focus the app. And the autofilled text is a noisy starting point — I usually delete it and write something specific anyway.

**Goal**: Replace the in-app pre-filled modal flow at EOD with a cleaner two-step:
1. Fire an **OS desktop notification** at 16:55 so the reminder reaches me even when Helm isn't focused.
2. Clicking the notification (or just focusing Helm afterward) **navigates to the Timesheet tab and opens the description editor for the EOD task with the description field empty**, ready for me to type. No autofill.

This piggybacks on Idea 1 — the running task must already be a clickable row in the Timesheet for this flow to have a target. So **Idea 2 depends on Idea 1**.

### Sketch of the approach

**Main process — `electron/scheduler/jobs.ts` (`eod-prompt.run()`)**:
- Drop `defaultText: titles.join('\n')` from the payload (or leave the field but pass `''`). Keep `taskTitles` available in case I want it later, but don't shove it into the editor.
- Change the event semantics for `kind: 'eod'` from "open modal" to "navigate + focus editor". A new `eventBus` event name like `eod-focus-entry` might be cleaner than overloading `description-prompt` — decide Monday. Carries `{ entryId, taskId }`.
- Fire `new Notification({...})` (Electron built-in, no new dependency):
  ```typescript
  // Pseudocode
  if (state.running && Notification.isSupported()) {
    const notif = new Notification({
      title: 'Helm — write your EOD description',
      body: `Open Helm and add a note for "${state.taskName}" before 4:59 PM.`,
    });
    notif.on('click', () => {
      const win = BrowserWindow.getAllWindows()[0];
      win?.show();
      win?.focus();
      // The renderer is already listening for the focus-entry event, so it
      // will switch tabs + open the editor on its own. Re-emit defensively
      // in case the click arrives after the original event was handled.
      timerBus.emit('eod-focus-entry', { entryId: state.entryId, taskId: state.taskId });
    });
    notif.show();
  }
  ```

**Renderer — `App.tsx` + `TimesheetEditor.tsx` + `TimeEntryDetail.tsx`**:
- Subscribe to the new `eod-focus-entry` event (forwarded over IPC from `timerBus` like the existing `description-prompt` channel).
- Handler:
  1. `setTab('timesheet')`
  2. `setSelectedEntryId('__running:' + taskId)` — the synthetic id from Idea 1
  3. After mount, `TimeEntryDetail` auto-focuses the description `<textarea>` (use a `ref` + `useEffect` keyed on `selectedEntryId` change with a "focus" intent flag).
- For `kind: 'eod'`, **do not** mount `DescriptionPrompt.tsx` modal. The Timesheet inline editor replaces it at EOD. Manual stops (`kind: 'manual'`) keep using the modal.
- Description field starts empty — no `taskTitles` prefill.

### Open question — running-entry editing

Idea 1 originally said "running entries are read-only since ClickUp's PUT on a running entry has been flaky." Idea 2 needs the **description** field to be editable on the running entry specifically. Two options to discuss Monday:

- **Option A**: Allow description edits on running entries via a focused PUT (`description` only, never `start`/`end`/`duration`). Test once before relying on it.
- **Option B**: Hold the typed description in renderer state (synthetic-entry-only buffer) and submit it as the description argument when `eod-stop` fires (or when the user manually stops). This avoids any PUT-on-running risk but requires a small main-process side: `stopTimer({ description })` already accepts a description, so this is mostly wiring.

Lean toward **Option B** — it sidesteps the ClickUp quirk entirely and matches the existing stop-with-description path. Idea 1's "running rows are read-only" guidance still holds for `start`/`end`/`duration`; just open description as a special case.

### Things to watch

- **Windows AppUserModelID**: Native notifications on Windows require `app.setAppUserModelId()` early in `electron/main.ts`. Check if it's wired; if not, add it (must match the NSIS installer's app id).
- **Skip when no timer is running**: Only fire the notification + focus event when `state.running`. Otherwise log `'skipped'` to `job_logs` per the "scheduler jobs must log even when they skip" rule in `CLAUDE.md`.
- **Modal removal isn't a regression for manual stops**: Keep `DescriptionPrompt.tsx` mounted for `kind: 'manual'`. Only the EOD path changes.
- **EOD-stop at 16:59 should still pick up whatever I typed**: If Option B, ensure the description in renderer state is flushed via `stopTimer({ description })` before the 16:59 main-process job auto-stops — otherwise the typed text is lost. Likely needs a renderer-side periodic save (debounce on every keystroke, ship to main via a tiny IPC `setRunningDescription(text)` that main holds in memory and uses when stopping).

### What likely changes

| Path | Change |
|---|---|
| `electron/scheduler/jobs.ts` | Drop `defaultText` autofill in `eod-prompt`. Add `Notification` call. Emit new `eod-focus-entry` event when running. |
| `electron/main.ts` | Verify/add `app.setAppUserModelId('com.helm.app')`. |
| `electron/preload.ts` + `electron/ipc/handlers.ts` + `shared/types.ts` | New IPC channel for `eod-focus-entry` forwarded to renderer. Plus `setRunningDescription(text)` if going Option B. |
| `electron/scheduler/timer.ts` | If Option B: hold `pendingRunningDescription` and pass into `stopTimer({ description })` automatically when scheduler-driven. |
| `src/App.tsx` | Subscribe to `eod-focus-entry`. On fire: switch tab, select synthetic running id. Suppress modal mount for `kind: 'eod'`. |
| `src/components/TimeEntryDetail.tsx` | Auto-focus description `<textarea>` when newly selected synthetic running entry. Allow description edits on running rows. |
| `src/components/DescriptionPrompt.tsx` | Drop the `kind: 'eod'` rendering path (or guard it off). Still handles manual stops. |

### Future-friendly (not 0.2.1)

- Settings toggle to disable the OS notification.
- Action buttons on the notification ("Snooze 5 min", "Open editor"). Native button support is fiddly cross-platform; defer.
- Restore the autofill as an opt-in setting if I miss it.

## Idea 3 — Bigger, centered current-task header

**Today's behavior**: `src/components/TimerBar.tsx` is a 56px (`h-14`) bar with the running task name on the left, "Fetch from ClickUp" / "Paused" indicators inline next to it, and elapsed time + Stop button on the right. The task name uses default font size (`text-base`-ish via the `[data-slot="timer-task"]` style). It's functional but visually de-emphasized — the most important thing on the screen reads as plain text.

**Goal**: Make the current-task name the visual centerpiece — slightly larger and horizontally centered in the bar — without losing the surrounding controls.

### Sketch of the approach

Three-column grid layout for the bar:

```
[left: indicator + Fetch button + Paused chip]  [center: task name (larger)]  [right: elapsed clock + Stop button]
```

- Switch the bar from `flex justify-between` to a 3-column CSS grid (`grid-cols-[1fr_auto_1fr]`) so the center column stays optically centered regardless of how wide the side columns get.
- Bump the task name from default to `text-lg` or `text-xl` (decide visually). Keep it on a single line, truncate with ellipsis if it overflows.
- Move the small "Current task" label *above* the name (already there as `[data-slot="timer-label"]`) and tighten its tracking so it reads as a kicker, not a peer of the name.
- Idle state (no timer): keep the centered layout but render `lexicon.noTimerLabel` in `text-inkMuted` so it doesn't shout.

### Things to watch

- Bar height may want a small bump (`h-14` → `h-16`) to breathe with the larger text. Any change here affects the resizable column geometry below — check `ResizableColumns.tsx` percentages still make sense.
- The "Paused: <task>" chip currently sits inline with the running task on the left. With the centered layout, move it under the centered task name as a smaller secondary line, or relocate to the left column.
- All visual — no IPC, no settings, no scheduler impact.

### What likely changes

| Path | Change |
|---|---|
| `src/components/TimerBar.tsx` | Grid layout, larger task font, idle-state parity. |
| `src/styles/index.css` | Possibly tweak `[data-slot="timer-task"]` / `[data-slot="timer-label"]` rules. |

## Idea 4 — Pull tasks from "Hugo's Laboratory" space

**Today's behavior**: `clickup.getAssignedTasks()` (`electron/services/clickup.ts:115`) pulls tasks from `/team/{workspaceId}/task` filtered by **`assignees[]=userId`** only. Tasks I'm not assigned to don't appear, even if they live in my personal space and I want to track time on them.

**Goal**: Surface tasks from my personal "Hugo's Laboratory" space alongside my assigned tasks — even when I'm not the assignee — so I can start timers on lab/exploratory work without first assigning it to myself in ClickUp.

### Sketch of the approach

ClickUp's `/team/{teamId}/task` accepts `space_ids[]` as a filter. Pull twice and merge:

1. Existing call: `assignees[]=userId` (unchanged).
2. New call: `space_ids[]={hugosLaboratorySpaceId}` (no `assignees[]`).
3. De-dupe on `task.id` (assigned tasks inside the space would otherwise show twice).

**Settings change** — generalize, don't hardcode "Hugo's Laboratory":
- New `Settings` field: `extraTaskSpaceIds: string[]` (array of space ids to pull regardless of assignee).
- Settings UI: a small input + add/remove list under "ClickUp" section. Need a way to look up the space id — either paste from ClickUp URL or list spaces via `GET /team/{id}/space` and let me pick.
- Default: empty array (no behavior change for fresh installs).

**Visual differentiation**:
- Tag tasks pulled via space-only (not assigned) with a subtle indicator — e.g. a small "Lab" pill next to the task name, or a different status-group color. Otherwise it's confusing why unassigned tasks appear.
- Optional: filter chip in `TaskFilterBar` to show/hide unassigned-but-in-extra-spaces tasks.

### Things to watch

- **Pagination doubles up**: each space query has its own pagination loop. Reuse the existing pagination wrapper in `getAssignedTasks` — extract into a helper if it'd be cleaner.
- **Task cache (`upsertCachedTasks`)**: should still receive both sets so timer name resolution works when starting a lab task.
- **Cache key for Hugo's Laboratory id**: don't hardcode the actual space id in source — keep it in `Settings`. The doc references "Hugo's Laboratory" by name only as a usage example.
- **API budget**: each extra space adds another paginated call. With 1-2 spaces it's fine; if I ever add many, consider parallelizing or rate-limiting.

### What likely changes

| Path | Change |
|---|---|
| `electron/services/clickup.ts` | New `getTasksInSpace(spaceId): Promise<Task[]>` (or extend `getAssignedTasks` to take optional space ids and merge). Add `listSpaces()` for the Settings picker. |
| `electron/ipc/handlers.ts` + `electron/preload.ts` + `shared/types.ts` | Expose `listSpaces()` to renderer for the Settings picker. |
| `electron/db/index.ts` + `shared/types.ts` | New `extraTaskSpaceIds: string[]` setting; default `[]`. |
| `src/components/Settings.tsx` | UI to view available spaces and toggle which are pulled. |
| `src/components/TaskList.tsx` | Optional "Lab" pill for unassigned-but-in-extra-space tasks. |

## Idea 5 — Custom UI-only status equivalences

**Today's behavior**: `TaskList.tsx` groups tasks by ClickUp's actual status string. My ClickUp workspace has multiple status sets (looks like ~2 distinct ones) — different lists use different vocab. So "In Design" in one list and "Scoping" in another show as **separate groups** in Helm even though they mean the same thing to me. The status dropdown in `TaskDetail.tsx` (added in 0.2.0 / 1a) writes back to ClickUp using the real status — that should not change.

**Goal**: Define **UI-only equivalence groups** that merge multiple raw ClickUp statuses into a single Helm-side group label. Purely cosmetic — the underlying ClickUp status is untouched. Fully user-configurable so I can declare "In Design" + "Scoping" → display as "Scoping", and so on.

### Sketch of the approach

**New `Settings` field**:
```typescript
statusEquivalences: {
  groupName: string;   // display label in Helm — e.g. "Scoping"
  members: string[];   // raw ClickUp status names — e.g. ["In Design", "Scoping"]
  color?: string;      // optional CSS color or hex; falls back to first member's status color
}[];
```

**Rendering logic in `TaskList.tsx`**:
- Build a lookup `Map<rawStatus, groupName>` from the equivalences config.
- When grouping tasks, replace each task's effective group key with `lookup.get(task.status) ?? task.status`. Group color = `equivalence.color ?? firstMember.statusColor`.
- The group's existing reorder (1b) and collapse (1c) state from 0.2.0 keys off the **display group name**, not the raw status — so toggling collapse on "Scoping" applies whether the underlying tasks are "In Design" or "Scoping" in ClickUp.

**Settings UI** (`src/components/Settings.tsx`):
- New section "Status equivalences".
- For each group: an input for the display name, a multi-select of all known raw statuses across all my lists (use `getListStatuses` from 1a, aggregated), and an optional color swatch.
- Add/remove buttons. Save on blur, debounced like other settings.

**Edit flow**:
- The status **dropdown in `TaskDetail.tsx`** keeps showing the **raw** ClickUp statuses for the current task's list (because that's what ClickUp accepts on PUT). It does NOT show equivalence group names — that distinction matters. The grouping is a list-view abstraction, not an edit-time abstraction.
- When the user picks a real status that is a member of an equivalence group, the task simply lands in the merged display group on next refresh.

### Things to watch

- **Order of operations vs 0.2.0 status dropdown (1a)**: Idea 5 depends on 1a being built (which it is, in 0.2.0). The status dropdown writes raw statuses; equivalences only affect the list view's grouping.
- **Discovery of all raw statuses**: across multiple lists, the same raw name can appear with different colors. Pick deterministically (first encountered? alphabetical? user-configured?). For 0.2.1, take the first encountered — rare clash, easy to fix manually via the optional `color` override.
- **Reorder/collapse persistence (from 0.2.0 1b/1c)**: keys must move from raw status to display group. This is a one-time migration in `getSettings()` — when reading the existing `taskStatusGroupOrder` and `collapsedStatusGroups`, leave them alone but treat them as display-group keys going forward. Worst case: pre-existing entries become slightly stale and the user re-toggles them. Acceptable.
- **No ClickUp writes**: triple-check no code path uses the equivalence group name as a status to PUT. The grouping logic is read-only over `task.status`.
- **Empty / no equivalences case**: behavior must be identical to today (raw statuses become group names 1:1).

### What likely changes

| Path | Change |
|---|---|
| `electron/db/index.ts` + `shared/types.ts` | New `statusEquivalences` setting + serialization. |
| `src/components/TaskList.tsx` | Apply equivalence map in the grouping `useMemo` before sorting. |
| `src/components/Settings.tsx` | New UI section to define groups (display name + member raw statuses + color). |
| `src/hooks/useListStatuses.ts` (from 0.2.0) | Add an aggregated "all known raw statuses across all my lists" selector for the Settings UI. |

### Future-friendly (not 0.2.1)

- Default presets (e.g. "Common dev workflow: Open / In Progress / Review / Done"). Skip — single user, easy to roll my own.
- Per-equivalence custom icon. Defer.

## Side note — scheduled jobs at 4:55 PM and 5 PM

Yes, they still fire. Two clarifications:

1. The actual times are **16:55 (4:55 PM)** for `eod-prompt` and **16:59 (4:59 PM)** for `eod-stop` — not 5:00 PM exactly. See the cron table in `CLAUDE.md`.
2. They run in the main process via `node-cron` (`electron/scheduler/index.ts`) using the timezone from `Settings`, completely independent of the renderer. Even if the timesheet UI is reworked or the window is closed (as long as Helm is running), they'll still fire and log to `job_logs`. Check Settings → "Recent job fires" to confirm after they run.
