# Release 0.3.1

Patch (or minor) release on top of `0.3.0`. Captures the **deferred subset** carved out of the original 0.3.0 plan — the ideas that didn't make the first cut because of scope. Everything here is **not ready to build**, will firm up when prioritized.

Bumped via `npm version patch` (or `minor` if Settings UI surface area expands enough) → `npm run release` when ready. See [WORKFLOW.md](./WORKFLOW.md) for the full release process.

> **Numbering note**: original idea numbers from the combined 0.3.0 plan are preserved (1, 2, 6, 7, 8) so cross-references with `docs/releases/0.3.0.md` and the original conversation stay intact. The gaps (3, 4, 5, 9) are because those four ideas shipped in 0.3.0.

---

## Why a separate release

The original 0.3.0 plan bundled nine ideas. Four of them (Ideas 3, 4, 5, 9) shipped in 0.3.0 because they were small/medium and immediately useful. The remaining five live here as 0.3.1 because either:

- They add new IPC surface or new Settings UI (Ideas 1, 2, 6, 8 — bigger touch).
- They're worth building but lower priority than the 0.3.0 set (Ideas 6, 7, 8).

Splitting them out keeps each release reviewable and avoids one chunky 0.3.0 release.

Two bugs surfaced after 0.3.0 manual verification — captured below as **Fix A** and **Fix B** so they ship alongside the new ideas.

## Fix A — Eliminate the timesheet range-switch delay

**Today's behavior**: Toggling between "Today" and "This week" in `TimesheetEditor` shows the previous range's data for ~200–800ms during the API roundtrip. The list briefly displays week data while loading today data (or vice versa). Cause: `useTimeEntries(timesheetRange)` re-fetches on every range change, leaving stale `entries` in state until the new response arrives.

Compounding cost: `App.tsx` already runs a parallel `useTimeEntries('week')` for the per-task badge — so we fetch the same week twice per session.

**Goal**: Range switch is instantaneous. No flash of stale data. One API call serves both the timesheet panel and the badge.

### Sketch of the approach

Drop the dual fetch. Keep only `useTimeEntries('week')` (rename to `allEntries` for clarity). Filter renderer-side for the timesheet panel:

```typescript
// Pseudocode in App.tsx
const allEntries = useTimeEntries('week'); // single source

const timesheetEntries = useMemo(() => {
  if (timesheetRange === 'today') {
    const cutoff = startOfToday();
    return allEntries.entries.filter((e) => e.start >= cutoff);
  }
  return allEntries.entries;
}, [allEntries.entries, timesheetRange]);

const mergedEntries = useMemo(
  () => mergeRunningEntry(timesheetEntries, timer, Date.now()),
  [timesheetEntries, timer, elapsedMs]
);
```

`taskTotals` keeps using `allEntries` directly (always week-based). `timesheet.save / .remove / .create / .load` calls become `allEntries.save / ...`. Same hook, same semantics.

Side benefit: switching is instant because the filter is pure renderer-side and the data is already in memory.

### What likely changes

| Path | Change |
|---|---|
| `src/App.tsx` | Drop the second `useTimeEntries`. Compute `timesheetEntries` via `useMemo` filter on `allEntries.entries`. Update all callers (`onSave`, `onCreate`, `onRefresh`, stop-handler `load()`). |

## Fix B — Per-task badge missing for some tasks

**Today's behavior**: Some tasks the user has tracked time on don't show a duration pill in `TaskList`. Most likely cause: ClickUp logs the time entry against a *subtask*, but the user views the *parent* task in the list. The parent's `id` doesn't match `entry.taskId`, so `taskTotals.get(parent.id)` is undefined and the pill renders nothing.

**Goal**: Parents roll up time logged on their subtasks. A parent task in the list shows total time = its own entries + all descendants' entries. Subtasks still show their own direct time independently.

### Sketch of the approach

In the `taskTotals` `useMemo`, walk up the parent chain for each entry and credit every ancestor:

```typescript
// Pseudocode
const taskById = new Map(tasks.map((t) => [t.id, t]));
const map = new Map<string, number>();
for (const e of merged) {
  if (!e.taskId) continue;
  if (e.start < cutoff) continue;
  const dur = e.duration || 0;
  let cursor: string | null = e.taskId;
  const seen = new Set<string>();
  while (cursor && !seen.has(cursor)) {
    seen.add(cursor);
    map.set(cursor, (map.get(cursor) || 0) + dur);
    cursor = taskById.get(cursor)?.parentId || null;
  }
}
```

Cycle guard via `seen` is defensive — task hierarchies should never cycle but data corruption could create one.

### Things to watch

- **Double-counting in aggregates**: a sum across all visible tasks now over-counts because parent + subtask both hold the same minutes. The badge is per-task only, so this only matters if we ever build a "total visible task time" indicator. Note for future stats panel (Idea 8): aggregate from raw entries, not from `taskTotals`.
- **Tasks not in `tasks`**: if `e.taskId` references a task that's not in `tasks` (archived, different list, not assigned), the cursor walk stops immediately because `taskById.get` returns undefined for the parent lookup. The leaf still gets credited; ancestors don't. Acceptable.
- **No taskList changes** — TaskList just consumes `taskTotals` as before.

### What likely changes

| Path | Change |
|---|---|
| `src/App.tsx` | Augment the `taskTotals` `useMemo` with the parent-chain walk. |

## Idea 1 — Pull tasks from "Hugo's Laboratory" space

*(Originally deferred from 0.2.1 → 0.3.0, now further deferred to 0.3.1.)*

**Today's behavior**: `clickup.getAssignedTasks()` (`electron/services/clickup.ts:115`) pulls tasks from `/team/{workspaceId}/task` filtered by **`assignees[]=userId`** only. Tasks I'm not assigned to don't appear, even if they live in my personal space and I want to track time on them.

**Goal**: Surface tasks from my personal "Hugo's Laboratory" space alongside my assigned tasks — even when I'm not the assignee — so I can start timers on lab/exploratory work without first assigning it to myself in ClickUp.

### Sketch of the approach

ClickUp's `/team/{teamId}/task` accepts `space_ids[]` as a filter. Pull twice and merge:

1. Existing call: `assignees[]=userId` (unchanged).
2. New call: `space_ids[]={hugosLaboratorySpaceId}` (no `assignees[]`).
3. De-dupe on `task.id` (assigned tasks inside the space would otherwise show twice).

**Settings change** — generalize, don't hardcode "Hugo's Laboratory":
- New `Settings` field: `extraTaskSpaceIds: string[]` (array of space ids to pull regardless of assignee).
- Settings UI: a small input + add/remove list under the "ClickUp" section. List spaces via `GET /team/{id}/space` and let me pick.
- Default: empty array (no behavior change for fresh installs).

**Visual differentiation**:
- Tag tasks pulled via space-only (not assigned) with a subtle "Lab" pill next to the task name. Otherwise it's confusing why unassigned tasks appear.
- Optional: filter chip in `TaskFilterBar` to show/hide unassigned-but-in-extra-spaces tasks.

### What likely changes

| Path | Change |
|---|---|
| `electron/services/clickup.ts` | Accept optional space ids in `getAssignedTasks` (or new `getTasksInSpace`). Add `listSpaces()` for the Settings picker. |
| `electron/ipc/handlers.ts` + `electron/preload.ts` + `shared/types.ts` | Expose `listSpaces()` to renderer. |
| `electron/db/index.ts` + `shared/types.ts` | New `extraTaskSpaceIds: string[]` setting; default `[]`. |
| `src/components/Settings.tsx` | UI to view available spaces and toggle which are pulled. |
| `src/components/TaskList.tsx` | Optional "Lab" pill for unassigned-but-in-extra-space tasks. |

## Idea 2 — Custom UI-only status equivalences

*(Originally deferred from 0.2.1 → 0.3.0, now further deferred to 0.3.1.)*

**Today's behavior**: `TaskList.tsx` groups tasks by ClickUp's actual status string. My ClickUp workspace has multiple status sets (~2 distinct ones); different lists use different vocab. So "In Design" in one list and "Scoping" in another show as **separate groups** in Helm even though they mean the same thing to me. The status dropdown in `TaskDetail.tsx` writes back to ClickUp using the real status — that should not change.

**Goal**: Define **UI-only equivalence groups** that merge multiple raw ClickUp statuses into a single Helm-side group label. Purely cosmetic — the underlying ClickUp status is untouched. Fully user-configurable.

### Sketch of the approach

**New `Settings` field**:
```typescript
statusEquivalences: {
  groupName: string;   // display label in Helm — e.g. "Scoping"
  members: string[];   // raw ClickUp status names — e.g. ["In Design", "Scoping"]
  color?: string;      // optional CSS color or hex; falls back to first member's status color
}[];
```

**Rendering logic in `TaskList.tsx`**:
- Build a lookup `Map<rawStatus, groupName>` from the equivalences config.
- When grouping tasks, replace each task's effective group key with `lookup.get(task.status) ?? task.status`. Group color = `equivalence.color ?? firstMember.statusColor`.
- The group's existing reorder (from 0.2.0 1b) and collapse (0.2.0 1c) state keys off the **display group name**, not the raw status — so toggling collapse on "Scoping" applies whether the underlying tasks are "In Design" or "Scoping" in ClickUp.

**Settings UI** (`src/components/Settings.tsx`):
- New section "Status equivalences".
- For each group: an input for the display name, a multi-select of all known raw statuses across all lists (use `getListStatuses`, aggregated), and an optional color swatch.
- Add/remove buttons. Save on blur, debounced like other settings.

**Edit flow**:
- The status **dropdown in `TaskDetail.tsx`** keeps showing the **raw** ClickUp statuses for the current task's list (because that's what ClickUp accepts on PUT). It does NOT show equivalence group names — that distinction matters.
- When the user picks a real status that is a member of an equivalence group, the task simply lands in the merged display group on next refresh.

### Things to watch

- **Reorder/collapse persistence** must move from raw status to display group keys. One-time migration: leave existing entries alone but treat them as display-group keys going forward. Worst case: pre-existing entries become slightly stale; user re-toggles. Acceptable.
- **No ClickUp writes**: triple-check no code path uses the equivalence group name as a status to PUT.
- **Empty / no equivalences case** must be identical to today (raw statuses become group names 1:1).
- **Interaction with pinned tasks (0.3.0 Idea 5)**: pinned tasks bypass status grouping entirely, so equivalences don't apply to them. Acceptable.

### What likely changes

| Path | Change |
|---|---|
| `electron/db/index.ts` + `shared/types.ts` | New `statusEquivalences` setting + serialization. |
| `src/components/TaskList.tsx` | Apply equivalence map in the grouping `useMemo` before sorting. |
| `src/components/Settings.tsx` | New UI section for groups (display name + member raw statuses + color). |
| `src/hooks/useListStatuses.ts` | Aggregate "all known raw statuses across all lists" selector for the Settings UI. |

## Idea 6 — Idle / lock detection

**Today's behavior**: If I lock my screen at 2pm with a timer running and unlock at 3pm, that hour gets logged on the running task whether I was working or at lunch. No detection of the idle gap.

**Goal**: When the OS reports the screen locked or the system idle for more than N minutes while a timer was running, on resume offer to truncate the entry to "when you locked." Default off-by-default for "Keep" so accidents don't lose time.

### Sketch of the approach

**Main process — `electron/main.ts` (or a new `electron/services/idle.ts`)**:
- Subscribe to `electron.powerMonitor` events:
  - `lock-screen` / `suspend` → record `idleStartedAt = Date.now()` in a module variable.
  - `unlock-screen` / `resume` → if `idleStartedAt` set and elapsed > `idleTimeoutMin` and a timer was running, emit a new `idle-truncate-prompt` event on `timerBus` with `{ idleStartedAt, idleEndedAt, taskId, taskName }`.
  - `system-idle` (long-form, fires after N seconds of no input) — works similarly but doesn't require lock.

**Renderer modal** (`src/components/IdleTruncatePrompt.tsx`):
- "You were away from {idleStartedAt} → {idleEndedAt} ({fmtDuration} idle). Was that work or a break?"
- Buttons:
  - **Truncate** → call new IPC `truncateRunningEntry(at: number)`. Main process: stop the running entry as if `clickup.stopTimer` happened at `idleStartedAt`. (Truncate-and-restart is a stretch — see Open questions.)
  - **Keep** → no-op. The entry continues uninterrupted.
  - **Snooze** → no-op for this lock event but re-prompt if it locks again.

**New `Settings`**:
- `idleDetectionEnabled: boolean` (default `true`).
- `idleTimeoutMin: number` (default `10`).
- `lockTriggersIdle: boolean` (default `true`) — whether `lock-screen` counts, since some users lock briefly without leaving their desk.

### Things to watch

- **Cross-platform behavior**: `lock-screen` on Windows fires reliably; macOS calls it `lock-screen` too via `powerMonitor`; Linux varies. Test on the user's actual platform first.
- **Race with `eod-stop`**: if the user is locked from 4:30 to 5:30, EOD already auto-stopped at 4:59. The idle prompt should detect this and not try to truncate an already-stopped entry. Check timer state before emitting.
- **Truncation actually requires two ClickUp ops** (stop with custom end, start new). Consider just stopping at `idleStartedAt` and not auto-restarting — let the user manually restart if they want. Simpler and less risky.

### What likely changes

| Path | Change |
|---|---|
| `electron/services/idle.ts` (new) | `powerMonitor` subscriber + state tracker. |
| `electron/main.ts` | Wire the idle service on app ready. |
| `electron/scheduler/timer.ts` | New `truncateRunningEntry(at)` (or similar) helper. Add `idle-truncate-prompt` to `TimerBus`. |
| `electron/preload.ts` + `electron/ipc/handlers.ts` + `shared/types.ts` | New IPC for the prompt event + truncate action. |
| `electron/db/index.ts` + `shared/types.ts` | New idle settings. |
| `src/components/IdleTruncatePrompt.tsx` (new) | The modal. |
| `src/App.tsx` | Subscribe to `onIdleTruncatePrompt`. |
| `src/components/Settings.tsx` | Toggle + threshold input. |

### Open questions

- Truncate-and-restart vs. truncate-only? Lean truncate-only — simpler, lower risk.
- Should the modal be modal-modal (blocking) or a non-blocking banner? Modal — easy to dismiss accidentally otherwise.
- Default threshold: 10 min, or higher? 10 catches most lunch breaks; 5 might be noisy. Start at 10.

## Idea 7 — Untracked-gap callout on the timeline

**Today's behavior**: `TimelineBar` renders entry segments and a lunch ghost band. If I forgot to start a timer for half an hour at 11am, the timeline just shows empty space — easy to miss.

**Goal**: Render hairline diagonal stripes during configured work hours where no entry covers that minute. Strict visual layer — no actions, no clicks; just awareness.

### Sketch of the approach

**New `Settings`**:
- `workHoursStart: number` (default `8 * 60` minutes, i.e. 8:00).
- `workHoursEnd: number` (default `17 * 60`, 17:00).

**In `TimelineBar.tsx`**:
- Compute the "covered" minute set from `resolved` (existing). Find the inverse during work hours.
- Render each gap as an absolutely-positioned `<div>` at the same vertical position as entry segments, with:
  ```
  background: repeating-linear-gradient(45deg, transparent 0 6px, rgb(var(--ink-muted) / 0.15) 6px 8px);
  pointer-events: none;
  ```
- Skip during the existing lunch band (already a ghost layer).
- `pointer-events: none` so they don't intercept clicks meant for entry segments.

### Things to watch

- **Don't render outside work hours** — that's normal "not at desk" time, not a gap. Important to avoid noise.
- **Running entries cover up to `now`** — gaps should only render *behind* the current minute, not ahead of it. The `TimelineBar` already projects running entries up to `now`; just clamp gap rendering to `Math.min(now, workHoursEnd)`.
- **Per-day work hours vs single block** — a single start/end is simpler. Different hours per weekday is over-engineering for now.
- **Multi-day TimelineBars (0.3.0 Idea 9)**: each per-day bar gets its own gap calculation; same logic, just iterated.

### What likely changes

| Path | Change |
|---|---|
| `electron/db/index.ts` + `shared/types.ts` | New `workHoursStart`/`workHoursEnd` settings. |
| `src/components/TimelineBar.tsx` | Compute gap intervals; render striped layer. |
| `src/components/Settings.tsx` | Two `<input type="time">` fields under a "Timeline" section. |

### Open questions

- Do gaps before the timer's first entry of the day count? Probably yes — "you didn't start tracking until 9:30 today" is useful awareness.
- Different stripe color in different themes? Use `--ink-muted` so it follows existing token system; probably fine across themes.

## Idea 8 — Weekly stats panel

**Today's behavior**: There's no aggregated view of where my time went. Tasks tab shows current state; Timesheet shows the entry log. Anything broader requires manual mental arithmetic.

**Goal**: New tab (`'stats'`) showing weekly bar charts of hours-per-status-group, hours-per-list, and hours-per-task. Pull data from `useTimeEntries('week')`. Nothing fancy.

### Sketch of the approach

**New tab**:
- Add `'stats'` to the `Tab` union in `App.tsx`.
- Tab button next to "Settings" (or replacing — single user, fewer tabs better; tbd).
- Renders a new component `src/components/StatsPanel.tsx`.

**Three views** (toggle at top of panel):
1. **By status group** — sum entries grouped by their task's status (using equivalences from Idea 2 if shipped).
2. **By list** — sum entries grouped by `task.listName`.
3. **By task** — top N tasks by hours, sorted descending.

**Bar chart implementation** — hand-rolled SVG, no charting library:
- Each row: label + horizontal bar + duration text.
- Bar width = `(taskHours / maxHours) * 100%`.
- Color from theme tokens (`--accent`) or, for status groups, the status color (already in cached tasks).
- 30 lines of JSX, zero new dependencies. A library would be overkill for what amounts to a stacked sequence of `<div>`s with widths.

**Data shape** — reuse `useTimeEntries('week').entries`, group/sum in a `useMemo`. Cached tasks (from `getCachedTask`) provide list name + status.

### Things to watch

- **Untracked entries** (no `taskId`) — bucket as "Untracked" in all three views.
- **Running entry** — synthetic from 0.2.1 — should be included with its current elapsed duration so the chart is live during the day.
- **Week boundary** — `useTimeEntries('week')` already does Monday-start (see `clickup.listTimeEntries`). Stats inherit that.

### What likely changes

| Path | Change |
|---|---|
| `src/components/StatsPanel.tsx` (new) | The whole panel — toggle + bar charts. |
| `src/App.tsx` | New `'stats'` tab. Mount `StatsPanel`. Pass `entries`, `tasks`. |
| `src/utils/stats.ts` (new, optional) | Pure aggregation helpers — easier to reason about than inline `useMemo`s. |

### Open questions

- Hand-rolled SVG vs. lightweight chart lib (e.g. `recharts`, `visx`)? Lean hand-rolled — bundle stays tiny, theme tokens just work, no dep maintenance.
- Make stats reactive to range (today/week/month)? Today is too short to be interesting; month would need a different `listTimeEntries` range. Start with week-only; iterate.
- Include a "by day" breakdown (Mon–Fri columns)? Probably yes for the by-task view — see which days I focused on what. Defer to a follow-up if it complicates the panel.
