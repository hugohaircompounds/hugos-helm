# Release 0.4.3

Patch on top of `0.4.2`. Two features:

1. **Edit the start time of the running timesheet entry** — fixes "I forgot to start the timer" without losing the running window.
2. **Explicit Save for time-entry descriptions** — visible Save button + status indicator on the description textarea; running-entry descriptions can finally land in ClickUp without stopping the timer first.

Bumped via `npm version patch` → `npm run release` when ready. See [WORKFLOW.md](./WORKFLOW.md) for the full release process.

---

## Idea 1 — Edit the start time of the running timesheet entry

**Today's behavior** (`src/components/TimeEntryDetail.tsx:147`): for a running entry, the Start input is `disabled={running}`. The only ways to correct a "I started this 90 minutes ago, not 5 minutes ago" mistake are:

- Stop the timer, edit the now-completed entry's `start`, restart the timer (loses the running context — duration is wrong, the new running window starts fresh).
- Fix it in ClickUp web manually.

**Goal**: the Start input is editable while the timer is running. On blur, the new start writes through to ClickUp and the timer-bar elapsed time + the synthetic running entry's `start` update within ~1s.

### Sketch of the approach

The synthetic running entry (`__running:{taskId}`) cannot be PUT directly — its id is renderer-only. Route through the timer service, which already knows the real ClickUp entry id (`timer_state.entryId`):

1. **New IPC** — `updateRunningEntryStart(start: number): Promise<TimerState>` on `HelmApi`.
2. **`electron/scheduler/timer.ts`** — new `updateRunningEntryStart(start)`:
   - Read `prev = getTimerState()`. Throw if `!prev.running` or `!prev.entryId`.
   - Reject non-finite or future starts.
   - `await clickup.updateTimeEntry(prev.entryId, { start })`.
   - `saveTimerState({ ...prev, startedAt: start })` and `emitChange(next)`.
3. **`src/components/TimeEntryDetail.tsx`** — remove `disabled={running}` from the Start input. In `onBlur`, branch on `running`:
   - Running → `window.helm.updateRunningEntryStart(v)`. On error, alert.
   - Not running → existing `onSave(entry.id, { start, duration })` path.

End stays `disabled={running}` — running entries don't have an end and editing the start doesn't change that.

### Things to watch

- **`startedAt` propagates downstream automatically.** `useTimer.ts` re-renders on `helm:timer-changed`; `mergeRunningEntry()` rebuilds the synthetic entry with the new start.
- **`entryId` is null briefly after a standup auto-resume** until `syncFromRemote` reconciles (60s). In that window, the IPC throws "No running timer to edit." Acceptable — narrow gap, user retries.
- **`datetime-local` minute granularity.** Resulting `start` is minute-aligned. Fine for human time-tracking.
- **EOD / idle-truncate race.** Both stop the timer (clearing `entryId`); a stop in flight surfaces as a 404 from ClickUp on the PUT, caught + alerted; the next `syncFromRemote` confirms ground truth.

## Idea 2 — Explicit Save for time-entry descriptions

**Today's behavior** (`src/components/TimeEntryDetail.tsx`): the description textarea is uncontrolled (`defaultValue={entry.description || ''}`) with two divergent save paths:

- **Non-running entry** — `onBlur` fires `onSave(entryId, { description })` → `clickup.updateTimeEntry`. Save is implicit; only blur triggers it.
- **Running entry** — `onChange` schedules a 300ms-debounced `setRunningDescription(text)` that buffers in `electron/scheduler/timer.ts`'s `pendingRunningDescription` module-level. The buffer flushes to ClickUp **only when the timer stops**. Until then, the typed text lives only in main's memory; ClickUp doesn't see it.

**Reported failure**: "I'm writing a description in the timesheet, sometimes it doesn't save what I typed, and when I click out it's not there anymore — making me retype." Two distinct loss modes match this:

1. **Running-entry mode**: user clicks running entry, types, switches tabs/clicks another entry → returns later → textarea shows `entry.description` from the loaded list, which is whatever was on ClickUp at the last fetch. The buffered draft in main is invisible to the renderer until the timer stops. Looks like data loss.
2. **Non-running blur race**: user types fast, clicks another entry quickly. Blur fires `onSave` (async) right as React unmounts the textarea. If the save errors silently or the entries-list refresh laps the user's view-switching, the description appears to revert.

**Goal**: An explicit `Save` button (+ `Cmd/Ctrl+Enter`) next to the description textarea that pushes immediately to ClickUp regardless of running/non-running. Visible feedback (`Saved · 14:32`) so the user knows the write landed. Implicit-on-blur save is **removed for the description field only** — start/end/duration keep their existing onBlur save (different semantics).

### Sketch of the approach

#### Running vs. non-running unified path

For both, "Save" pushes the current textarea contents to ClickUp via `PUT /time_entries/{id}`:

- **Non-running** — `entry.id` is a real ClickUp time-entry id. Reuse `clickup.updateTimeEntry` via the existing `onSave` callback.
- **Running** — `entry.id` is the synthetic `__running:{taskId}` id. Route through the timer service:
  - New IPC `flushRunningDescription(text: string): Promise<void>`. Main reads `getTimerState().entryId`, PUTs description, writes the same text into `pendingRunningDescription` so the next stop-flush is idempotent.
  - To make the saved value visible across navigations, also expose the running entry's last-known description via `getRunningDescription(): Promise<string>` so `TimeEntryDetail` can seed its initial text correctly when re-mounting onto the running entry. Without this the synthetic entry's `description: ''` (from `buildRunningEntry`) wins on every remount.

#### UI surface

Below the description textarea, right-aligned:

```
[ description textarea ......................... ]
                                Unsaved changes  [Save]
```

After a successful save:

```
[ description textarea ......................... ]
                                  Saved · 14:32  [Save]
```

**Status indicator states**:
- `Saved · {time}` (idle, no pending changes; button disabled)
- `Unsaved changes` (textarea differs from last-saved; button enabled)
- `Saving…` (IPC in flight; button disabled)
- `Save failed: {err}` (write errored; button re-enabled, contents preserved)

`Cmd/Ctrl+Enter` while focused → triggers Save. Plain Enter still inserts a newline.

Save button is disabled when text equals last-saved.

#### Implicit blur-save: REMOVED for the description field only

The Save button (or `Cmd+Enter`) is the only path to push the description to ClickUp from a non-running entry. Rationale: more predictable, no silent-failure mode where blur fires during a tab switch and the user loses certainty about whether it landed. `start`, `end`, `duration` keep their existing `onBlur → onSave` flow.

#### Running-entry debounced buffer stays

For running entries, `onChange` continues to feed `pendingRunningDescription` via the debounced + flush-on-blur `setRunningDescription` IPC. **No change to that path.** It's the safety net: even if the user never clicks Save, stopping the timer still flushes their typed text. The Save button is the explicit "I want this in ClickUp now without stopping the timer" path.

#### Switching entries with unsaved changes

By removing implicit blur-save, this is now a real loss path. `TimeEntryDetail` exposes an `onDirtyChange(dirty: boolean)` callback; `App.tsx` keeps a ref tracking it. Before changing `selectedEntryId`, if `dirtyRef.current === true`, prompt with `confirm("Discard unsaved description changes?")`. Cancel → stay on the current entry; OK → switch.

A `beforeunload` listener wires the same prompt for window close / refresh.

#### Things to watch

- **Running entry: timer-stop overlap** — if the user clicks Save while EOD also fires `stopTimer` (which flushes the buffer), the two writes race. `flushRunningDescription` writes the same value into `pendingRunningDescription` before returning, so the subsequent stopTimer flush is an idempotent re-PUT of the same description.
- **Cold-start with a remote running timer that has a description** — when `syncFromRemote` finds an in-progress entry on ClickUp, also seed `pendingRunningDescription` with `remote.description ?? ''`. Otherwise the buffer is empty until the user types and the come-back textarea seed via `getRunningDescription` shows nothing.
- **Synthetic id detection** — `isRunningId(entry.id)` exists. Branch the Save action between `onSave` (non-running) and `flushRunningDescription` (running).
- **No cursor jumps** — the textarea is controlled but only updates from `onChange` (user input) and from the entry-id-change reseed. No external value writes during typing.

### What likely changes

| Path | Change |
|---|---|
| `shared/types.ts` | Add `flushRunningDescription(text: string): Promise<void>` and `getRunningDescription(): Promise<string>` to `HelmApi`. |
| `electron/preload.ts` | Wire both. |
| `electron/ipc/handlers.ts` | Register `timer:flushRunningDescription` and `timer:getRunningDescription`. |
| `electron/scheduler/timer.ts` | Add `flushRunningDescription(text)`. Seed `pendingRunningDescription` with `entry.description` on `startTimer` and `syncFromRemote` so come-back reads the right value. Existing `getRunningDescription()` accessor stays. |
| `src/components/TimeEntryDetail.tsx` | Description textarea becomes controlled. Track `lastSaved` / `dirty` / `saveStatus` / `savedAt`. On entry-id change, seed via `getRunningDescription` for running entries, `entry.description` for non-running. Add Save button + status indicator + `Cmd/Ctrl+Enter`. Remove `onBlur → onSave` for description (start/end onBlur stays). Notify parent of dirty changes via `onDirtyChange` prop. |
| `src/App.tsx` | Maintain a `dirtyRef`; wrap `setSelectedEntryId` with a confirm guard; add a `beforeunload` listener that fires the same prompt. |
